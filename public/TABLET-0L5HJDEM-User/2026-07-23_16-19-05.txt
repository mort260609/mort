  Receive vs. process are on different threads. OkHttp invokes onMessage on its own dedicated WebSocket reader thread. handleMessage immediately hands off by launch-ing a coroutine, so the OkHttp reader thread returns quickly and continues reading frames. Parsing (gson.fromJson) and all processData work run inside the launched coroutine, NOT on the OkHttp thread.
  
  Processing is serialized on a single-threaded dispatcher. The scope uses Dispatchers.IO.limitedParallelism(1) (line 41) — effectively a single-thread executor. Therefore messages are processed strictly one at a time, in the order OkHttp delivered/launched them.

There is no bounded Channel or queue with backpressure here — the "queue" is the unbounded set of pending coroutines waiting for the single dispatcher thread. Under a burst, work accumulates as queued coroutines rather than blocking the reader.

  What is already correct
  On both clients the socket reader thread is not blocked by processing — it enqueues and returns immediately, so bytes keep coming off the wire regardless of how backed up processing is. That half of your requirement is already met.
  
  What is NOT correct — the real issue
  Processing is single-threaded and strictly FIFO on both clients. res_code 2 ticks arrive continuously, and each tick does non-trivial work on that one lane:
  
  full JSON decode of the fat WsMessageResponse (Gson / JSONDecoder),
  per-position repricing,
  chart-candle folding.
  Because every message — ticks and discrete responses alike — shares one lane, a discrete response the user is waiting on (res_code 7 trade result, 4 account, 3 chart history, 5 history) sits behind every tick already queued and only runs after they drain. Under a tick burst the lane backs up and latency for those responses grows. That is exactly the coupling you want to eliminate: res_code 2 processing currently does affect processing of other responses (though not their receipt).
  
  Swift has a second, milder coupling: both ticks and every other handler mutate GlobalState under one shared NSRecursiveLock (GlobalState.swift:107).
  
  Recommendation: split into two lanes by res_code
  Give res_code 2 its own processing lane, separate from a "control" lane that handles everything else. Since a tick can be identified from the raw string cheaply — Swift already does text.contains("\"res_code\":2") at WebSocketManager.swift:856, and Kotlin can do the same — you route before the expensive parse:
  
  Control lane (serial): res_code 1,3,4,5,6,7,8,120–122,201,610,202,611 — order preserved, never delayed by ticks.
  Tick lane (separate): res_code 2 only. Ticks are latest-wins by nature, so this lane can additionally coalesce/drop under backlog instead of processing every stale tick.
  Concretely:
  
  Swift — add a second DispatchQueue (e.g. tickQueue, can be .userInitiated), route in handleMessage on the cheap res_code:2 check.
  Kotlin — add a second scope, e.g. tickScope = CoroutineScope(Dispatchers.Default.limitedParallelism(1)), route in handleMessage before gson.fromJson.

  Ordering of a positions snapshot (res_4) vs. tick repricing (res_2). They now run on different threads; both touch openPositions/symbolPositions.
