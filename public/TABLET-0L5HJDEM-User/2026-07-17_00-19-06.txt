  New file auth.js — stateless HMAC tokens (payload + signature, 30-day expiry), issued at /login and /register. I chose stateless over a session table for two reasons: your schema.sql is stale (it lacks reset_token and trial_ends_at, which the code already uses) and nothing applies it, so a table would have meant a migration you didn't ask for; and unlike the admin panel's in-memory Map (api.js:17), these survive a restart, which mobile clients need. I unit-tested tampered payloads, tampered signatures, wrong-secret signing, expiry, and malformed headers — all rejected.
  
  
  
  Set AUTH_TOKEN_SECRET in .env before deploying. Without it the server signs with a random per-boot key and logs a warning; tokens then die on every restart. Clients recover silently, but it's needless churn.
  
  
  
  Clients send the account, and carry the token
  
  All three now pass currentAccount.login at both report call sites, and attach Authorization: Bearer … at their single HTTP choke point (api.ts:44, ApiClient.kt, APIClient.swift).
  
  
  
  Since tokens expire, each client now recovers from a 401 by re-issuing from its cached credentials and retrying once — Kotlin via an OkHttp Authenticator, Swift via an actor-gated refresh, web via a shared promise.   This breaks already-installed apps. Any client that hasn't shipped these changes sends no token and will get a 401 on /report. That's inherent to closing an auth hole, but it means server and clients must ship together, or /report breaks for everyone on the old build.

  All REST APIs are now protected
  
  requireAuth went from "verify signature" to verify → resolve the user from the DB → overwrite email in both query and body with the authenticated one (auth.js:74-104).
