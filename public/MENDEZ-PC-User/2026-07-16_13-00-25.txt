  Fix Typing Indicator & Deleted Chat Revival
  I've completed a fresh audit of the code (messaging.ts, [chatId].tsx, and the recent SQL migrations like 011_revive_deleted_chat_on_message.sql) to understand the current architecture.
  
  Here is what I've found regarding the two bugs.
  
  Root Cause Analysis
  1. Stuck / Missing Typing Indicator
  The logic introduced to manage typing channels has two critical race conditions:
  
  Missing Indicator: If you start typing immediately when the chat opens, broadcastTyping skips sending the true state because the Supabase channel is still in the JOINING state (takes ~200-500ms). However, the local isTypingRef.current is set to true. Subsequent keystrokes see that it's already true and never attempt to broadcast again, so your typing indicator never appears for the other person.
  Stuck Indicator: If you hit "Send" and immediately back out of the chat, the component unmounts. In the cleanup function, it calls channel.untrack() (async) and supabase.removeChannel(channel) (sync/detached). Tearing down the local channel immediately can abort the pending untrack() or track(false) WebSocket messages, leaving your state stuck as true on the server.
  2. Message Not Displaying After Re-opening a Deleted Chat
  When you create a new chat with a user you previously deleted a chat with, createDirectChat correctly finds the old chat ID and routes you to it. However, when [chatId].tsx loads, it fetches the chat metadata. I strongly suspect that your live RLS policies (likely on the chats table) are filtering out rows where chat_members.deleted_at IS NOT NULL. If the chats query returns null, the chat state becomes null. When you type a message and press "Send", the handleSend function silently returns early (if (!chat) return;). The message is never sent, the optimistic UI doesn't run, and the 011 migration trigger never gets a chance to revive the chat.
