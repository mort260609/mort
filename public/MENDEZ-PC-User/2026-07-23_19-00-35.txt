  # Migrate VotingFreeze + NexaiGovStaking Admin to Timelock
  
  Per Dmitry's review: `VotingFreeze`'s `ADMIN_ROLE` (controls who can
  create proposals) is still on the deployer EOA — only `NexaiCore` got
  migrated via `MigrateAdminToTimelock.s.sol`. Same gap likely applies
  to `NexaiGovStaking` if it has any admin-gated functions. Fix both,
  following the exact same pattern already proven for `NexaiCore`.
  
  ## Scope
  
  - **`VotingFreeze.sol`**: `ADMIN_ROLE` (gates `createProposal`) should
    move to the `TimelockController`, same as `NexaiCore`'s
    `ADMIN_ROLE`/`DEFAULT_ADMIN_ROLE`
  - **`NexaiGovStaking.sol`**: check whether it has any admin-gated
    functions at all — recall it was deliberately built with minimal
    access control (`distributeRewards` is intentionally permissionless,
    per its own contract comment). If there's no real `ADMIN_ROLE` to
    migrate, say so rather than inventing one to migrate for
    consistency's sake
  
  ## Requirements
  
  - New script, e.g. `script/MigrateGovContractsToTimelock.s.sol` —
    separate from `MigrateAdminToTimelock.s.sol` (which is scoped to
    `NexaiCore` and already reviewed/approved), not merged into it
  - **Same grant-before-revoke ordering** as the original migration
    script: grant the timelock the new role(s) first, verify the grant
    landed, only then revoke the EOA's role — so the EOA can never
    self-lock-out mid-migration if an earlier step silently failed
  - Reuse the existing `TimelockController` address (from
    `DeployTimelock.s.sol`) — do not deploy a second timelock instance
  - Same testnet-only scope, same style/conventions as the existing
    migration script

    after migration
  - Confirm the timelock (via the same Safe → Timelock → execute flow
    already proven in `NexaiCoreAdminTest`) can successfully call
    `createProposal` post-migration
  - If `NexaiGovStaking` genuinely has no admin role to migrate, no test
    needed there — just confirm and document that finding
  
  ## Notes
  
  - Testnet only, no mainnet
  - Standard process: run `forge build && forge test` before reporting

  pragma solidity ^0.8.20;
  
  import {Test} from "forge-std/Test.sol";
  import {NexaiGovToken} from "../src/governance/NexaiGovToken.sol";
  import {VotingFreeze} from "../src/governance/VotingFreeze.sol";

  /**
   * @title VotingFreezeAdminTest
   * @dev End-to-end tests for script/MigrateGovContractsToTimelock.s.sol - proves
   * VotingFreeze's ADMIN_ROLE/DEFAULT_ADMIN_ROLE actually move from the deployer EOA to
   * the timelock, and that the timelock can exercise that role through the real
   * Safe(2-of-3) -> Timelock(48h) -> execute flow already proven in
   * NexaiCoreAdminTest. Forks Base Sepolia and calls Safe's real, canonical v1.4.1
   * singleton/factory, same "genuine contract, not a mock" reasoning as
   * NexaiCoreAdmin.t.sol. Requires BASE_SEPOLIA_RPC_URL in `.env`.
   *
   * NexaiGovStaking is deliberately not covered here - it has no AccessControl/admin
   * role at all (see MigrateGovContractsToTimelock.s.sol's doc comment), so there is
   * nothing for a migration test to prove for it.
   */
  contract VotingFreezeAdminTest is Test {
      address constant SAFE_SINGLETON = 0x29fcB43b46531BcA003ddC8FCB67FFE91900C762;
      address constant SAFE_PROXY_FACTORY = 0x4e1DCf7AD4e460CfD30791CCC4F9c8a4f820ec67;
      address constant COMPATIBILITY_FALLBACK_HANDLER = 0xfd0732Dc9E303f09fCEf3a7388Ad10A83459Ec99;
  
      NexaiGovToken public govToken;
      VotingFreeze public votingFreeze;
  
      address public adminSafe; // 2-of-3, proposer on the timelock
      TimelockController public timelock;
  
      uint256 constant SIGNER_KEY_1 = 0xA11CE1;
      uint256 constant SIGNER_KEY_2 = 0xA11CE2;
      uint256 constant SIGNER_KEY_3 = 0xA11CE3;
  
      address public deployer = address(this);
  
      function setUp() public {
          vm.createSelectFork(vm.envString("BASE_SEPOLIA_RPC_URL"));
  
          // --- Token + VotingFreeze, mirrors DeployGovToken.s.sol / DeployVotingFreeze.s.sol ---
          govToken = new NexaiGovToken();
          votingFreeze = new VotingFreeze(address(govToken));
          govToken.setCapExempt(address(votingFreeze), true);
  
          // --- Real Safe, via the canonical factory already deployed on this fork,
          // mirrors DeploySafes.s.sol (only AdminSafe is needed here - VotingFreeze has
          // no guardian/pause concept) ---
          address[] memory owners = new address[](3);
          owners[0] = vm.addr(SIGNER_KEY_1);
          owners[1] = vm.addr(SIGNER_KEY_2);
          owners[2] = vm.addr(SIGNER_KEY_3);
  
          adminSafe = _deploySafe(owners, 2, 0);
  
          // --- Timelock, mirrors DeployTimelock.s.sol ---
          address[] memory proposers = new address[](1);
          proposers[0] = adminSafe;
          address[] memory executors = new address[](1);
          executors[0] = address(0); // permissionless execution
  
          timelock = new TimelockController(48 hours, proposers, executors, address(this));
          timelock.renounceRole(timelock.TIMELOCK_ADMIN_ROLE(), address(this));
  
          // --- Migrate VotingFreeze admin control, mirrors MigrateGovContractsToTimelock.s.sol ---
          votingFreeze.grantRole(votingFreeze.DEFAULT_ADMIN_ROLE(), address(timelock));
          votingFreeze.grantRole(votingFreeze.ADMIN_ROLE(), address(timelock));
          votingFreeze.revokeRole(votingFreeze.ADMIN_ROLE(), deployer);
          votingFreeze.revokeRole(votingFreeze.DEFAULT_ADMIN_ROLE(), deployer);
      }
  
      // ------------------------------------------------------------------
      // 1. Old EOA is locked out of createProposal after migration
      // ------------------------------------------------------------------
      function test_EOACannotCreateProposalAfterMigration() public {
          vm.expectRevert();
          votingFreeze.createProposal(7);
      }
  
      // ------------------------------------------------------------------
      // 2. Full Safe(2-of-3) -> Timelock(48h) -> execute flow can create a proposal
      // ------------------------------------------------------------------
      function test_FullSafeToTimelockToExecute_CreatesProposal() public {
          bytes memory createCalldata = abi.encodeWithSelector(VotingFreeze.createProposal.selector, uint256(7));
  
          bytes32 salt = bytes32(uint256(1));
          bytes32 id = _scheduleViaAdminSafe(address(votingFreeze), createCalldata, salt);
          assertTrue(timelock.isOperationPending(id), "not pending after schedule");
  
          vm.warp(block.timestamp + 48 hours + 1);
          timelock.execute(address(votingFreeze), 0, createCalldata, bytes32(0), salt);
  
          assertTrue(timelock.isOperationDone(id), "operation not marked done");
          assertEq(votingFreeze.proposalCount(), 1, "proposal was not actually created");
          (uint256 endTime,,) = votingFreeze.proposals(0);
          assertEq(endTime, block.timestamp + 7 days, "proposal created with wrong duration");
      }
  
      // ------------------------------------------------------------------
      // Helpers - same pattern as NexaiCoreAdminTest
      // ------------------------------------------------------------------
  
      function _deploySafe(address[] memory owners, uint256 threshold, uint256 saltNonce) internal returns (address) {
          bytes memory initializer = abi.encodeWithSelector(
              ISafe.setup.selector,
              owners,
              threshold,
              address(0),
              bytes(""),
              COMPATIBILITY_FALLBACK_HANDLER,
              address(0),
              uint256(0),
              payable(address(0))
          );
          address safe = ISafeProxyFactory(SAFE_PROXY_FACTORY).createProxyWithNonce(SAFE_SINGLETON, initializer, saltNonce);
          assertEq(ISafe(safe).getThreshold(), threshold, "deployed Safe has unexpected threshold");
          return safe;
      }
  
      /// @dev Schedules `data` targeted at `target` through AdminSafe (2-of-3), the
      /// timelock's sole proposer, using the timelock's own minimum delay.
      function _scheduleViaAdminSafe(address target, bytes memory data, bytes32 salt) internal returns (bytes32 id) {
          bytes memory scheduleCalldata = abi.encodeWithSelector(
              TimelockController.schedule.selector, target, uint256(0), data, bytes32(0), salt, timelock.getMinDelay()
          );
          uint256[] memory keys = new uint256[](2);
          keys[0] = SIGNER_KEY_1;
          keys[1] = SIGNER_KEY_2;
          bool ok = _execSafeTx(adminSafe, keys, address(timelock), scheduleCalldata);
          assertTrue(ok, "AdminSafe schedule failed");
          id = timelock.hashOperation(target, 0, data, bytes32(0), salt);
      }
  
      /// @dev Executes `data` targeted at `to` through `safe`, signed by exactly the
      /// given signer keys.
      function _execSafeTx(address safe, uint256[] memory signerKeys, address to, bytes memory data)
          internal
          returns (bool success)
      {
          bytes memory signatures = _buildSignatures(safe, signerKeys, to, data);
          try ISafe(safe).execTransaction(to, 0, data, 0, 0, 0, 0, address(0), payable(address(0)), signatures) returns (
              bool ok
          ) {
              success = ok;
          } catch {
              success = false;
          }
      }
  
      /// @dev Builds a Safe-format signature blob: signs the transaction's EIP-712 hash
      /// with each given key, ordered by increasing signer address (a hard Safe
      /// requirement for its signature check).
      function _buildSignatures(address safe, uint256[] memory signerKeys, address to, bytes memory data)
          internal
          view
          returns (bytes memory signatures)
      {
          uint256 txNonce = ISafe(safe).nonce();
          bytes32 txHash =
              ISafe(safe).getTransactionHash(to, 0, data, 0, 0, 0, 0, address(0), address(0), txNonce);
  
          address[] memory signers = new address[](signerKeys.length);
          for (uint256 i = 0; i < signerKeys.length; i++) {
              signers[i] = vm.addr(signerKeys[i]);
          }
          for (uint256 i = 1; i < signerKeys.length; i++) {
              uint256 keyToInsert = signerKeys[i];
              address addrToInsert = signers[i];
              uint256 j = i;
              while (j > 0 && signers[j - 1] > addrToInsert) {
                  signers[j] = signers[j - 1];
                  signerKeys[j] = signerKeys[j - 1];
                  j--;
              }
              signers[j] = addrToInsert;
              signerKeys[j] = keyToInsert;
          }
  
          for (uint256 i = 0; i < signerKeys.length; i++) {
              (uint8 v, bytes32 r, bytes32 s) = vm.sign(signerKeys[i], txHash);
              signatures = abi.encodePacked(signatures, r, s, v);
          }
      }
  }

  pragma solidity ^0.8.20;
  
  import {Script, console} from "forge-std/Script.sol";
  import {VotingFreeze} from "../src/governance/VotingFreeze.sol";
  
  /**
   * @title MigrateGovContractsToTimelock
   * @dev ONE-WAY migration of VotingFreeze's admin control from the deployer EOA to the
   * existing TimelockController (DeployTimelock.s.sol) - same reasoning and same
   * grant-then-verify-then-revoke ordering as MigrateAdminToTimelock.s.sol, kept as its
   * own script rather than merged into that one since it's scoped to a different
   * contract (VotingFreeze, not NexaiCore) and that script is already reviewed/approved.
   * Reuses the existing timelock instance - does not deploy a second one.
   *
   * Order matters and is chosen to avoid ever self-locking the EOA out mid-migration:
   *   1. Grant TIMELOCK the new roles first
   *   2. Verify each grant actually landed
   *   3. Only then revoke the EOA's roles, ADMIN_ROLE before DEFAULT_ADMIN_ROLE - the EOA
   *      needs DEFAULT_ADMIN_ROLE (the role-admin for ADMIN_ROLE) to still be able to fix
   *      anything if an earlier step were wrong, so it's the last thing given up.
   *
   * NexaiGovStaking was checked and deliberately does NOT inherit AccessControl at all -
   * stake/unstake/claim are gated by the caller only ever moving their own funds, and
   * distributeRewards is intentionally permissionless per its own contract doc comment.
   * There is no admin role to migrate there; nothing for this script to do for it, not an
   * oversight.
   * Testnet only.
   */
  contract MigrateGovContractsToTimelockScript is Script {
      // Set these to the addresses printed by DeployVotingFreeze.s.sol / DeployTimelock.s.sol.
      address constant VOTING_FREEZE = address(0);
      address constant TIMELOCK = address(0);
  
      function run() external {
          require(VOTING_FREEZE != address(0), "Set VOTING_FREEZE to DeployVotingFreeze.s.sol's address first");
          require(TIMELOCK != address(0), "Set TIMELOCK to DeployTimelock.s.sol's TimelockController address first");
  
          VotingFreeze votingFreeze = VotingFreeze(VOTING_FREEZE);
  
          uint256 deployerPrivateKey = vm.envUint("PRIVATE_KEY");
          address deployer = vm.addr(deployerPrivateKey);
          vm.startBroadcast(deployerPrivateKey);

  # NEXAI Website — Implementation Spec
  
  **Purpose of this document:** hand-off spec for Claude Code (or Google Antigravity) to execute the NEXAI website build phase by phase. Each phase ends in a reviewable, deployable checkpoint. Do not start a phase until the prior one is reviewed and approved.
  
  ---
  
  ## 1. Project Context
  
  - **NEXAI** is a dual-token protocol on **Base** (Ethereum L2):
    - `$NEXAI` — onchain AI-sector index token, backed by a basket of 20 AI crypto projects
    - Governance/fundraising token (name TBD) — ERC20, fixed-window freeze during voting, staking-based fee reward layer (independent of voting weight), Sablier-based cancelable vesting streams
  - Smart contracts extend the **Reserve Protocol (DTF framework)**, written in Solidity.
  - Rebalancing is quarterly via DAO vote.
  - **Already deployed to Base Sepolia (testnet):** `NexaiIndexToken`, `NexaiCore`, mock WETH/USDC. End-to-end mint/redeem confirmed working through a trial frontend.
  - Governance is linear token-weighted voting (not quadratic).
  - **TODO before Phase 0 starts:** insert current deployed contract addresses (testnet, and mainnet once available) into `/config/contracts.ts`. Do not fabricate or assume addresses — pull from actual deployment records.
  
  ---
  
  ## 2. Tech Stack
  
  | Concern | Recommendation |
  |---|---|
  | Framework | Next.js (App Router) — SSR helps guest mode load fast/cached, better SEO |
  | Wallet connect | wagmi + viem + RainbowKit |
  | Email/social login | Web3Auth or Magic.link — **open decision, see Section 11** |
  | Styling | Tailwind CSS |
  | Animation | Framer Motion (spring/elastic UI transitions) + GSAP ScrollTrigger (scroll-driven chart/timeline work) |
  | Price chart | TradingView Lightweight Charts (handles blur/focus + marker overlays well) |
  | Holder count / indexing | Subgraph (The Graph) or Alchemy/Covalent — do **not** query holder count via direct RPC calls, it won't scale |
  | Tx history | Basescan API (Base only for now) |
  | AI assistant (basket tab) | Claude API, scoped per-token context (see Phase 3) |
  | Hosting | Vercel (pairs naturally with Next.js + guest-mode caching) |
  
  ---
  
  ## 3. Suggested Repo Structure
  
  ```
  /app
    /page.tsx                 → main landing (guest + authed states)
    /profile
    /basket
    /governance
  /components
    /landing (hero stat, chart, join-cta)
    /sidebar
    /profile-panel
  /lib
    /contracts.ts             → ABIs + addresses, imported not hardcoded inline
    /indexer.ts                → subgraph/Alchemy queries
    /web3auth.ts
  /config
    /contracts.ts              → TODO: fill in real addresses
  /hooks
    /useHolderCount.ts
    /useTokenBalances.ts
    /useTxHistory.ts
  ```
  
  ---
  
  ## 4. Phase 0 — Foundation
  
  **Goal:** deployable skeleton, fast.
  
  - Repo scaffold (Next.js + Tailwind + wagmi/RainbowKit wired in)
  - Wallet connect (MetaMask via RainbowKit) working against Base Sepolia
  - Guest mode: static landing page served from cache, no live RPC/API calls — this is the DDoS-resistant public entry point
  - Deploy pipeline to Vercel (staging URL)
  
  **Acceptance criteria:** staging URL live, wallet connects, guest mode loads with zero live queries (verify via network tab).
  
  ---
  
  ## 5. Phase 1 — Core Landing Experience
  
  **Goal:** the main screen, without scroll polish yet.
  
  - Live holder-count stat, backed by subgraph/indexer, refreshed on an interval (e.g. every 2–5 min) — not real-time RPC
  - "Join Now" CTA → routes into auth flow
  - Price chart rendered (static state, no scroll effects yet) using Lightweight Charts
  - AI Grants Fund stat block — **data source TBD, see Section 11**; build as an admin-editable field for now, not assumed on-chain
  
  **Acceptance criteria:** landing page shows real holder count and chart with live price data; Join Now routes correctly.
  
  ---
  
  ## 6. Phase 2 — Auth + Profile
  
  **Goal:** users can log in multiple ways and see their own data.
  
  - Web3Auth or Magic.link integration for email/Gmail login (embedded wallet)
  - Profile panel (top-right "i" icon):
    - `$NEXAI` + governance token balances (wagmi reads against deployed contracts)
    - Convert button — **scope explicitly per Section 11 before building**; at minimum, USDC ↔ NEXAI via existing `NexaiCore` mint/redeem
  - History button: compact on-chain tx scan, Base only, via Basescan API
  
  **Acceptance criteria:** logged-in user sees correct live balances and their real tx history; guest mode is unaffected.
  
  ---
  
  ## 7. Phase 3 — Sidebar Tabs
  
  **Goal:** Basket and Governance tabs.
  
  - **Basket tab:** detail view for all 20 basket tokens (more thorough than a typical exchange listing). Ship with curated static write-ups per token first. Layer in the AI assistant afterward:
    - AI assistant scoped strictly to the token being viewed (pass token context into the prompt, don't build an open-ended chatbot)
    - Uses Claude API; keep this behind a feature flag so it can ship independently of the rest of the tab
  - **Governance tab:** build the UI against mock data now. Do not wire to real governance contracts until the governance token (in progress separately) is deployed — this tab will show "coming soon" until then.
  
  **Acceptance criteria:** Basket tab shows accurate static info for all 20 tokens; AI assistant answers scoped questions correctly; Governance tab renders cleanly with mock data.
  
  ---
  
  ## 8. Phase 4 — Scroll Polish + Convert Finalization
  
  **Goal:** the signature scroll experience, and convert button fully scoped/built.
  
  - On scroll down: stats + Join Now fly up (spring/elastic, Framer Motion), chart transitions blur → sharp, milestone markers appear, timeline pans right with further scroll
  - On scroll up: timeline reverses, stats spring back to origin
  - On max scroll: social links (Telegram, X) appear — persistent across every tab
  - Parallax between chart layer and text layer
  - **Mobile:** do not port the same scroll-jacking mechanism directly — build a simplified fade/stack version for small viewports (see Section 12)
  - Finalize Convert button per the Section 11 decision (may include DEX aggregator integration for NEXAI ↔ governance token conversion if approved)
  
  **Acceptance criteria:** scroll experience matches spec on desktop; mobile equivalent is smooth (test on actual devices, not just resized browser); convert button executes real transactions on testnet.
  
  ---
  
  ## 9. Phase 5 — Deferred
  
  Not part of this build push — sequence after Phase 4 ships and is reviewed:
  
  - Multichain tx history (beyond Base)
  - Real governance data (once governance contracts are live)
  - TVL counter
  - Persistent "Buy NEXAI" CTA
  - Any further mobile-first refinement beyond the Phase 4 baseline
  
  ---
  
  ## 10. Non-Functional Requirements
  
  - **Mobile-first:** every phase should be checked on mobile viewport, not just desktop — flagged explicitly because Phase 4's parallax/scroll-jacking is desktop-first by nature.
  - **Performance:** guest mode must load with no live RPC/API calls (cache-served). Animation-heavy sections should degrade gracefully (reduced motion for users with `prefers-reduced-motion`).
  - **Security:** no private keys or seed phrases ever touch the frontend or logs. Wallet interactions go through wagmi/viem standard flows only. Web3Auth/Magic.link keys handled per their SDK's documented security model — do not roll custom key storage.
  
  ---
  
  ## 11. Open Decisions Needed Before Building (flag to Dmitry)
  
  These are called out because building them wrong is expensive to unwind — confirm before Claude Code implements:
  
  1. **Web3Auth vs. Magic.link** — pick one; affects embedded wallet UX and integration work.
  2. **Convert button scope** — USDC↔NEXAI and NEXAI↔basket-token can route through the existing `NexaiCore` mint/redeem. Converting **between the two NEXAI tokens** (index ↔ governance) is not native — it needs either a dedicated swap pool or a DEX aggregator (0x/1inch) integration. Confirm whether this is in scope for this push or deferred.
  3. **AI Grants Fund data source** — on-chain (needs a grants registry contract, not yet built) or off-chain (admin-editable field)? Determines Phase 1 scope.
  4. **Basket-tab AI assistant** — confirm Claude API is acceptable for this (vs. another provider), and confirm token-level context/data source for the curated write-ups.
  
  ---
  
  ## 12. Review Checkpoints
  
  Each phase should be reviewed (by Mendez, then Dmitry) before the next begins. Recommend Claude Code stops and flags for review at the end of every phase rather than running straight through to Phase 4.

  # NEXAI Website — Implementation Spec
  
  **Purpose of this document:** hand-off spec for Claude Code (or Google Antigravity) to execute the NEXAI website build phase by phase. Each phase ends in a reviewable, deployable checkpoint. Do not start a phase until the prior one is reviewed and approved.
  
  ---
  
  ## 1. Project Context
  
  - **NEXAI** is a dual-token protocol on **Base** (Ethereum L2):
    - `$NEXAI` — onchain AI-sector index token, backed by a basket of 20 AI crypto projects
    - Governance/fundraising token (name TBD) — ERC20, fixed-window freeze during voting, staking-based fee reward layer (independent of voting weight), Sablier-based cancelable vesting streams
  - Smart contracts extend the **Reserve Protocol (DTF framework)**, written in Solidity.
  - Rebalancing is quarterly via DAO vote.
  - **Already deployed to Base Sepolia (testnet):** `NexaiIndexToken`, `NexaiCore`, mock WETH/USDC. End-to-end mint/redeem confirmed working through a trial frontend.
  - Governance is linear token-weighted voting (not quadratic).
  - **TODO before Phase 0 starts:** insert current deployed contract addresses (testnet, and mainnet once available) into `/config/contracts.ts`. Do not fabricate or assume addresses — pull from actual deployment records.
  
  ---
  
  ## 2. Tech Stack
  
  | Concern | Recommendation |
  |---|---|
  | Framework | Next.js (App Router) — SSR helps guest mode load fast/cached, better SEO |
  | Wallet connect | wagmi + viem + RainbowKit |
  | Email/social login | Web3Auth or Magic.link — **open decision, see Section 11** |
  | Styling | Tailwind CSS |
  | Animation | Framer Motion (spring/elastic UI transitions) + GSAP ScrollTrigger (scroll-driven chart/timeline work) |
  | Price chart | TradingView Lightweight Charts (handles blur/focus + marker overlays well) |
  | Holder count / indexing | Subgraph (The Graph) or Alchemy/Covalent — do **not** query holder count via direct RPC calls, it won't scale |
  | Tx history | Basescan API (Base only for now) |
  | AI assistant (basket tab) | Claude API, scoped per-token context (see Phase 3) |
  | Hosting | Vercel (pairs naturally with Next.js + guest-mode caching) |
  
  ---
  
  ## 3. Suggested Repo Structure
  
  ```
  /app
    /page.tsx                 → main landing (guest + authed states)
    /profile
    /basket
    /governance
  /components
    /landing (hero stat, chart, join-cta)
    /sidebar
    /profile-panel
  /lib
    /contracts.ts             → ABIs + addresses, imported not hardcoded inline
    /indexer.ts                → subgraph/Alchemy queries
    /web3auth.ts
  /config
    /contracts.ts              → TODO: fill in real addresses
  /hooks
    /useHolderCount.ts
    /useTokenBalances.ts
    /useTxHistory.ts
  ```
  
  ---
  
  ## 4. Phase 0 — Foundation
  
  **Goal:** deployable skeleton, fast.
  
  - Repo scaffold (Next.js + Tailwind + wagmi/RainbowKit wired in)
  - Wallet connect (MetaMask via RainbowKit) working against Base Sepolia
  - Guest mode: static landing page served from cache, no live RPC/API calls — this is the DDoS-resistant public entry point
  - Deploy pipeline to Vercel (staging URL)
  
  **Acceptance criteria:** staging URL live, wallet connects, guest mode loads with zero live queries (verify via network tab).
  
  ---
  
  ## 5. Phase 1 — Core Landing Experience
  
  **Goal:** the main screen, without scroll polish yet.
  
  - Live holder-count stat, backed by subgraph/indexer, refreshed on an interval (e.g. every 2–5 min) — not real-time RPC
  - "Join Now" CTA → routes into auth flow
  - Price chart rendered (static state, no scroll effects yet) using Lightweight Charts
  - AI Grants Fund stat block — **data source TBD, see Section 11**; build as an admin-editable field for now, not assumed on-chain
  
  **Acceptance criteria:** landing page shows real holder count and chart with live price data; Join Now routes correctly.
  
  ---
  
  ## 6. Phase 2 — Auth + Profile
  
  **Goal:** users can log in multiple ways and see their own data.
  
  - Web3Auth or Magic.link integration for email/Gmail login (embedded wallet)
  - Profile panel (top-right "i" icon):
    - `$NEXAI` + governance token balances (wagmi reads against deployed contracts)
    - Convert button — **scope explicitly per Section 11 before building**; at minimum, USDC ↔ NEXAI via existing `NexaiCore` mint/redeem
  - History button: compact on-chain tx scan, Base only, via Basescan API
  
  **Acceptance criteria:** logged-in user sees correct live balances and their real tx history; guest mode is unaffected.
  
  ---
  
  ## 7. Phase 3 — Sidebar Tabs
  
  **Goal:** Basket and Governance tabs.
  
  - **Basket tab:** detail view for all 20 basket tokens (more thorough than a typical exchange listing). Ship with curated static write-ups per token first. Layer in the AI assistant afterward:
    - AI assistant scoped strictly to the token being viewed (pass token context into the prompt, don't build an open-ended chatbot)
    - Uses Claude API; keep this behind a feature flag so it can ship independently of the rest of the tab
  - **Governance tab:** build the UI against mock data now. Do not wire to real governance contracts until the governance token (in progress separately) is deployed — this tab will show "coming soon" until then.
  
  **Acceptance criteria:** Basket tab shows accurate static info for all 20 tokens; AI assistant answers scoped questions correctly; Governance tab renders cleanly with mock data.
  
  ---
  
  ## 8. Phase 4 — Scroll Polish + Convert Finalization
  
  **Goal:** the signature scroll experience, and convert button fully scoped/built.
  
  - On scroll down: stats + Join Now fly up (spring/elastic, Framer Motion), chart transitions blur → sharp, milestone markers appear, timeline pans right with further scroll
  - On scroll up: timeline reverses, stats spring back to origin
  - On max scroll: social links (Telegram, X) appear — persistent across every tab
  - Parallax between chart layer and text layer
  - **Mobile:** do not port the same scroll-jacking mechanism directly — build a simplified fade/stack version for small viewports (see Section 12)
  - Finalize Convert button per the Section 11 decision (may include DEX aggregator integration for NEXAI ↔ governance token conversion if approved)
  
  **Acceptance criteria:** scroll experience matches spec on desktop; mobile equivalent is smooth (test on actual devices, not just resized browser); convert button executes real transactions on testnet.
  
  ---
  
  ## 9. Phase 5 — Deferred
  
  Not part of this build push — sequence after Phase 4 ships and is reviewed:
  
  - Multichain tx history (beyond Base)
  - Real governance data (once governance contracts are live)
  - TVL counter
  - Persistent "Buy NEXAI" CTA
  - Any further mobile-first refinement beyond the Phase 4 baseline
  
  ---
  
  ## 10. Non-Functional Requirements
  
  - **Mobile-first:** every phase should be checked on mobile viewport, not just desktop — flagged explicitly because Phase 4's parallax/scroll-jacking is desktop-first by nature.
  - **Performance:** guest mode must load with no live RPC/API calls (cache-served). Animation-heavy sections should degrade gracefully (reduced motion for users with `prefers-reduced-motion`).
  - **Security:** no private keys or seed phrases ever touch the frontend or logs. Wallet interactions go through wagmi/viem standard flows only. Web3Auth/Magic.link keys handled per their SDK's documented security model — do not roll custom key storage.
  
  ---
  
  ## 11. Open Decisions Needed Before Building (flag to Dmitry)
  
  These are called out because building them wrong is expensive to unwind — confirm before Claude Code implements:
  
  1. **Web3Auth vs. Magic.link** — pick one; affects embedded wallet UX and integration work.

  3. **AI Grants Fund data source** — on-chain (needs a grants registry contract, not yet built) or off-chain (admin-editable field)? Determines Phase 1 scope.
  4. **Basket-tab AI assistant** — confirm Claude API is acceptable for this (vs. another provider), and confirm token-level context/data source for the curated write-ups.
