  # Fix Cost Estimate: Real Live Quote Instead of Approximation
  
  Priority #1 per Dmitry. Currently `App.tsx` estimates the zapper's USDC
  cost using the basket weight ratio as a stand-in price
  (`estimatedSwapCostUsdc = requiredWeth * (usdcWeight / wethWeight)`).
  This was a known simplification — accurate only while the seeded pool's
  price matches the original ratio, and it's already drifted after real
  swaps (see the 3%→10% buffer bump earlier). Replace it with a real
  on-chain price quote.
  
  ## Approach: Uniswap V3 QuoterV2
  
  Base Sepolia QuoterV2 address: `0xC5290058841028F1614F3A6F0F5816cAd0df5E27`
  
  QuoterV2's `quoteExactOutputSingle` is the read-only counterpart to the
  swap function the zapper already calls — same parameters, but simulates
  the swap and returns the real expected input amount instead of actually
  executing it. This gives an accurate, current-price estimate without
  needing to guess from basket weights.
  
  ## Requirements
  
  - Add the QuoterV2 address to `constants.ts`, alongside a minimal ABI
    slice for `quoteExactOutputSingle` (same pattern as the existing
    `ZAPPER_ABI` — just the one function needed)
  - In `App.tsx`, replace `estimatedSwapCostUsdc`'s calculation: instead
    of the weight-ratio approximation, call `quoteExactOutputSingle` via
    `useReadContract` (or fold into the existing `useReadContracts` batch)
    with `tokenIn: USDC, tokenOut: WETH, amount: requiredWeth, fee: 3000`
  - Note: `quoteExactOutputSingle` in QuoterV2 is technically a
    non-view function under the hood (it reverts with the result encoded,
    a known Uniswap quirk) — confirm wagmi's `useReadContract` /
    `useSimulateContract` handles this correctly for QuoterV2 specifically;
    if `useReadContract` doesn't work cleanly, `useSimulateContract` is
    the correct wagmi hook for this exact pattern
  - Keep the same refresh cadence as the rest of the balance/allowance
    reads (5s polling) so the estimate stays reasonably fresh as the user
    is looking at the page
  - The 10% buffer on `maxUsdcIn` can likely be reduced now that the
    estimate itself is accurate (e.g. back to 3-5%) — but this is a
    judgment call, flag your reasoning rather than just picking a number
  - Keep the existing UI layout/display as-is — just swap what feeds the
    "~X USDC" number, no visual changes needed

  Here's what changed and why:
  
  Why useSimulateContract, not useReadContract. QuoterV2's quoteExactOutputSingle is declared nonpayable in the real contract — it internally calls the pool's swap(), which triggers a callback that computes the amounts and deliberately reverts so no tokens actually move, then the outer function catches that revert and decodes the numbers out of it before returning normally. Since Solidity won't let a function call something non-view and still be marked view itself — even though nothing ends up persisted — the ABI has to say nonpayable. useReadContract is typed to only accept view/pure ABI entries, so it rejects this at the type level. useSimulateContract doesn't have that restriction; it just runs an eth_call-style simulation regardless of declared mutability, which is exactly what a quote needs.
  
  Why it's a separate hook, not folded into the useReadContracts batch. Same reason — that batch hook is built on the same view/pure-only primitive, so a nonpayable quote function can't ride along in it. I did fold in one extra read to the batch, though: swapFee() — since NexaiZapper.swapFee is a public immutable already on-chain, reading it there means the 3000 fee tier isn't duplicated as a magic number in the frontend, and it'll stay correct automatically if the deployed zapper ever changes fee tiers.
  
  The buffer: 10% → 5%. The old 10% was compensating for the estimate model being wrong (stale weight ratio). Now the base number is a live quote, so the only thing left to buffer against is ordinary movement in the gap between "last quote refresh" (polled every 5s) and "tx actually lands" (wallet confirmation + block time — realistically tens of seconds). 5% is my judgment call for a testnet pool, where swings can be sharper than a real liquid market; flagging it as adjustable rather than treating it as settled.
  
  Still outstanding: the local forge test run on the zapper contract itself — unchanged by this frontend work, still needs to happen on your end.

  They absolutely **could** use Zoom or Google Meet.
  
  The fact that they're using Relay doesn't automatically mean it's a scam. Lots of startups use different tools:
  
  * Zoom
  * Google Meet
  * Microsoft Teams
  * Around
  * Riverside
  * Relay
  * Custom internal tools
  
  What matters is **how** they're asking you to use it.
  
  A few things stand out to me:
  
  **Normal startup explanation:**
  
  * They already use Relay internally.
  * Relay records meetings, generates transcripts, summaries, action items, etc.
  * They want all hiring conversations in one system.
  
  **Less normal explanation:**
  
  * They insist you install a desktop app when a browser version exists.
  * They won't switch to Meet or Zoom even when you're having issues.
  * They keep pushing you to download files from links they provide.
  
  The fact that they're asking for screenshots of errors isn't itself suspicious. If I were helping someone troubleshoot a meeting tool, I'd ask for screenshots too.
  
  What I'd do in your position:
  
  Ask casually:
  
  > "Out of curiosity, does Relay have a browser version? My setup seems to be having issues with the desktop client."
  
  or
  
  > "If Relay keeps failing, would Google Meet be okay as a backup?"
  
  A legitimate team will usually say:
  
  > "Sure, here's a Meet link."
  
  or
  
  > "Use the browser version."
  
  A scammer often becomes weirdly insistent that you install *their* software.
  
  One more thing: from everything you've shown me about Merklio so far, the conversations were unusually detailed for a typical crypto-job scam. They discussed your experience, smart contract security, salary expectations, and then moved you to management review. That's more effort than most scams put in.
  
  But the **software installation step is exactly where you should become more cautious.**
  
  Can you tell me:
  
  * Did Relay come from their official website (`relay.lc`)?
  * Or did they send you a direct download file/link through Telegram?
  
  That detail makes a huge difference.

  Fair points on the general principle — different companies do use different tools, and by itself that's not damning. But two things in your message actually cut the other way, not toward reassurance.
  
  First: the "connection failed" screen wasn't a real error, it's a manufactured one. It's the second half of that dialog that matters — "the desktop app uses a native network stack that bypasses browser WebRTC restrictions entirely," paired with a big red download button. Legitimate apps don't manufacture a fake technical failure to create urgency for you to grab an executable. That's a scripted trigger, not a bug report. So the test you're proposing ("ask if Meet would work as backup") is worth trying, but even a "sure, use Meet" answer wouldn't erase the fact that the app already tried to strong-arm you into the .exe once.
  
  Second: the sophistication argument runs backwards. The research on this exact campaign (Meeten/Meetio, tracked since 2024) specifically notes attackers use AI to build convincing sites, detailed cover stories, and even real-seeming documents to establish trust before the app request — precisely because a thin scam gets caught fast. Discussing your background, smart contract security, and salary in depth isn't evidence against the scam; it's consistent with the documented MO of that specific campaign, which explicitly targets people with Web3/crypto-adjacent skills. That your interview touched on smart contract security is, if anything, a match to the target profile, not a point in the company's favor.
  
  To your direct question: the download came from relay.lc itself, the official-looking site — not a random Telegram file. That doesn't help either; the whole point of the "AI-generated legitimate-looking company" approach is that the malware gets served from what looks like an official source, not a sketchy one.
  
  I'd still run your Meet/Zoom test if you want a clean signal, but I wouldn't take a "yes" as full clearance given the dialog you already saw. And regardless of the answer, don't run the Windows download.
