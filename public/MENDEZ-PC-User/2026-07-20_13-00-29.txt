  Root cause identified and fixed in index.tsx:231-245: the realtime effect for public-chat-members/public-messages-list created its channels unconditionally on mount, before myUserId was populated, then re-created them when myUserId landed. That churn hits a real race in @supabase/realtime-js — client.channel(topic) reuses any existing channel object sharing the same topic string, and client.removeChannel() is async while React's effect cleanup is fire-and-forget, so the second .subscribe() call can land on a channel that isn't cleanly closed yet and silently fail to (re)join with the correct filter. This is the same class of bug the typing indicator already had fixed via a if (!chatId || !myUserId) return; guard — that guard just never got applied here.
