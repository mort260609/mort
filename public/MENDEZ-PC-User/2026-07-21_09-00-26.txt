  # Phase 3, Round 3.4 — Block/Report, Search, Profiles, In-Chat Menu
  
  ## Context
  Rounds 3.1-3.3 are complete and verified on-device: typing indicators, read receipts, reactions, editing, deletion (for me/for everyone), pin/mute/archive, forwarding, reply-to-message, delete-chat, and a full UI overhaul are all working. Reference the current ARCHITECTURE.md (please generate a fresh one first if it's stale) before starting, and do the same "audit the real codebase and live database first" discipline established in the last handoff — don't assume any document is 100% accurate to current state.
  
  ## Scope — build exactly this
  
  ### 1. Block / Report
  - Add a `blocks` table: `blocker_id`, `blocked_id`, `created_at`
  - Add a `reports` table: `reporter_id`, `reported_user_id`, `message_id` (nullable — for reporting a specific message vs. a user generally), `reason`, `details`, `created_at`, `status` (default 'pending')
  - Blocking a user: hides their messages from you going forward, prevents them from starting new direct chats with you, removes you from each other's contact-matched suggestions
  - Report flow: simple custom in-app form (reason dropdown + optional details text field) — accessible from a user's profile view (see #3 below) or a long-press on a message
  - Blocked users should not be notified they've been blocked (silent to them)
  - Remember our standing rule: all UI must be custom in-app components, never native Alert/prompt
  
  ### 2. Search — in-chat and global
  - **In-chat search**: a search icon in the open chat screen that filters/highlights matching messages in the current conversation. This searches already-decrypted messages client-side (content is encrypted server-side and can't be searched via a Postgres query directly).
  - **Global search**: accessible from the chat list, searches across all conversations by decrypting and matching locally, plus matches on chat/group names and participant display names. Results should clearly indicate which chat each match came from.
  
  ### 3. Viewable profile for other users
  - Tapping a chat's header (avatar/name) opens the other person's profile view: display name, avatar, username, a media/shared-content placeholder (real content once Phase 4 exists), and Block/Report actions (from #1 above)
  - For groups: tapping the header opens group info (member list, etc. — check if this already exists from earlier group chat work, extend if needed)
  
  ### 4. In-chat overflow menu
  - A three-dot menu inside an open conversation (separate from the existing chat-list long-press menu), with:
    - "View Contact/Profile" — links to #3 above
    - "Media & Links" — placeholder/basic implementation until Phase 4 has real media
    - "Clear Chat" — deletes all messages in this conversation for the current user only (distinct from "Delete Chat," which removes the whole conversation from the list; Clear Chat empties the messages but keeps the conversation itself)
  
  ## Explicitly out of scope for this round
  - Media sharing itself (Phase 4) — Media & Links menu item can be a placeholder for now
  - Push notifications (Phase 5)
  - Status (Phase 6)
  
  ## Acceptance criteria
  - [ ] Blocking a user prevents them from messaging you and hides their existing messages from your view going forward
  - [ ] Reporting a user or message saves correctly to the reports table with the right reason/details
  - [ ] In-chat search correctly finds and highlights matching messages in the current conversation
  - [ ] Global search returns correct results across multiple chats, clearly attributed to the right conversation
  - [ ] Tapping a chat header opens a real profile view with correct data for that user
  - [ ] The in-chat overflow menu works, including a functioning "Clear Chat" that empties messages without deleting the conversation itself
  - [ ] No regressions to any previously verified functionality (typing, receipts, reactions, editing, deletion, pin/mute/archive, forwarding, reply)
  
  ## Testing requirement
  Same standard as every previous round: verify actual behavior on real devices (or your best available means) before reporting anything complete — show real evidence, not just code descriptions. Given the project's history, please explicitly flag anything you cannot personally verify (e.g. real-time behavior across two devices) rather than assuming it works.

  Round 3.4 — Block/Report, Search, Profiles, In-Chat Overflow Menu
  Context
  Rounds 3.1–3.3 are verified on-device. This round adds four features on top of that base: block/report, in-chat + global search, a viewable profile screen for other users, and an in-chat overflow menu (including "Clear Chat"). Per the project's established discipline (and this round's explicit instruction), the codebase and live DB were audited before designing anything — and that audit surfaced real drift worth fixing as part of this round, not just building the four features blind.
  
  Audit findings (why the plan looks the way it does)
  ARCHITECTURE.md is badly stale — dated Phase 3.1 (2026-07-07), predates reactions, editing, deletion, pin/mute/archive, forwarding, reply, the typing-indicator rewrite, and the blocks/reports tables entirely. Will regenerate at the end from actual current state.
  blocks and reports tables already exist live in Supabase (confirmed via direct SQL queries against the linked project) but exist in zero local migration files — supabase migration list shows local/remote in sync through 012, so these were created directly against the DB (dashboard SQL editor), completely outside version control. Schema is sound and matches the round's spec closely:
  blocks(blocker_id uuid, blocked_id uuid, created_at timestamptz default now()), composite PK (blocker_id, blocked_id), FKs to auth.users cascade. RLS: blocks_select/blocks_insert/blocks_delete all scoped to blocker_id = auth.uid() — a user can only see/manage blocks they created, never see who has blocked them (correctly supports "blocked users should not be notified").
  reports(id uuid pk default gen_random_uuid(), reporter_id, reported_user_id, message_id nullable → messages(id) on delete set null, reason text not null, details text, status text default 'pending', created_at). RLS: reports_select/reports_insert scoped to reporter_id = auth.uid().
  Neither table is RLS-safe for the actual product requirement yet — no reporter can moderate, and nothing stops a blocked user's messages from still writing to messages. Fixed below.
  Zero client code references either table (grepped, no hits). src/types/database.ts has no Block/Report types.
  group-info.tsx already fully exists and works (name edit, add/remove members, admin toggle, leave group) — group-info is not greenfield, just needs the header wiring to point at it correctly (already does, for groups) and a couple of native Alert.alert() calls in it should be swapped to custom modals while touched (violates the standing no-native-alert rule).
  [chatId].tsx's header tap (navigateToInfo) only handles group chats today — disabled={!isGroup} on the direct-chat case, i.e. tapping a 1-on-1 chat's header currently does nothing. This is exactly the gap Feature 3 needs to fill.
  No search infrastructure exists anywhere — the chat list's existing "Search chats..." bar (app/(app)/chats/index.tsx) is a pure client-side .includes() match on chat title only. No message-content search, no dedicated search screen, no reusable SearchBar component.
  message_deletions RLS (with_check: user_id = auth.uid()) already supports exactly what "Clear Chat" needs — bulk-inserting one row per message for the current user is just "delete for me" applied to every message in the chat, reusing the existing mechanism that fetchMessages() already filters against. No new schema needed for Clear Chat.
  handleScrollToMessage(msgId) already exists in [chatId].tsx (index lookup + scrollToIndex) — reusable for deep-linking from global search results into the exact message.
  User-confirmed UX decisions
  In-chat search: highlight + jump between matches (messages stay in place, up/down arrows, "n/m" counter) — not a filtered/hidden list.
  Global search: dedicated search icon → new full-screen search screen, separate from the existing chat-list title filter.
  Scoping decisions (stated explicitly, not silently narrowed)
  Block enforcement is done at the message-send layer, not a separate chat-creation gate. A blocked user can still technically create the chats/chat_members rows, but every messages INSERT is rejected by RLS if sender and recipient have a block between them (either direction) in a direct chat. Practically equivalent to "can't start a chat" — they can never get a message through — with far less complexity than also gating chat_members INSERT (which would need to reason about the other row being inserted in the same statement).
  "Hides their messages from you" is implemented as a full, permanent client-side filter (all messages from a blocked sender, past and future, hidden from chat view + list preview + unread counting), not time-bounded to "only future messages." Simpler and matches the plain reading of "hides their existing messages... going forward."
  Contact-suggestion hiding is one-directional in this round: blocked users are filtered out of your own matched-contacts results (via match-contacts edge function, using the caller's own RLS-visible blocks rows). The reverse (you disappearing from their suggestions) would require the edge function to bypass RLS with the service-role key to check "has this candidate blocked the caller" — real added risk/complexity for a secondary requirement not covered by this round's acceptance criteria.
