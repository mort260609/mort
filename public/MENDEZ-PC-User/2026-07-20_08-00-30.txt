  # NEXAI Contracts
  
  Onchain AI-sector index protocol. Deposit approved tokens, mint $NEXAI
  representing a proportional share of the underlying basket, redeem
  $NEXAI any time for your share back.
  
  **Status**: Trial phase complete on Base Sepolia testnet. Basket
  currently WETH/USDC (placeholder) — real 20-token AI basket is planned
  for M2 (see `/docs/M2_Basket_Scaling_Design.pdf`).
  
  ---
  
  ## Architecture
  
  ### Core Contracts
  
  | Contract | Purpose |
  |---|---|
  | `NexaiIndexToken.sol` | The $NEXAI ERC-20. Mint/burn restricted to `CORE_ROLE`. |
  | `NexaiCore.sol` | Deposit/redeem logic. In-kind: exact basket token amounts in, $NEXAI out, and reverse. No swaps, no oracle reads — fixed-ratio math only. |
  | `PriceOracle.sol` | Chainlink + RedStone price facade. Inherited by `NexaiCore` for RedStone's calldata mechanism, but not currently used by mint/redeem math (reserved for future valuation/display features). |
  | `NexaiZapper.sol` | One-click single-token deposits. Swaps the portion needed to cover basket legs, then deposits into Core. Supports USDC, WETH, USDT entry — see "Zapper" below. |
  
  ### Design Principle: In-Kind Core, Convenience on Top
  
  `NexaiCore` deliberately has zero swap logic and zero oracle reads in
  its mint/redeem path. This keeps the fund-holding contract's attack
  surface minimal — no price feed can be stale, wrong, or manipulated if
  it's never read during a mint/redeem. All swap complexity lives in
  `NexaiZapper`, which never holds funds beyond a single transaction.
  
  This mirrors Reserve Protocol's actual DTF architecture, which NEXAI
  is modeled on.
  
  ### Zapper: Three Entry Paths
  
  - `zapDepositUsdc(nexaiAmount, maxUsdcIn)` — direct USDC leg + swap for WETH leg
  - `zapDepositWeth(nexaiAmount, maxWethIn)` — direct WETH leg + swap for USDC leg
  - `zapDepositUsdt(nexaiAmount, maxUsdtIn)` — USDT isn't a basket asset, so **two swaps**, no direct leg
  
  All three: pull the max budget upfront, swap via Uniswap V3
  (`SwapRouter02`, `exactOutputSingle`), deposit into Core, forward
  minted $NEXAI to the caller, refund unspent budget. Zero custody
  between transactions.
  
  **Guard**: `_basketIsExactlyTwoTokens()` probes `basketTokens(2)` and
  uses the expected out-of-bounds revert to detect a 3rd basket token —
  prevents a silent partial-swap failure if the basket is ever expanded
  without updating the zapper.

  ## Local Development
  
  ### Prerequisites
  - Foundry (`forge`, `cast`, `anvil`) — install via WSL if on Windows,
    Foundry doesn't support native Windows
  - `.env` file with `PRIVATE_KEY` and `BASE_SEPOLIA_RPC_URL` (never
    commit this file)
  
  ### Build & Test
  ```bash
  forge build
  forge test                          # full suite
  forge test --match-contract NexaiZapperTest -vvv   # one suite, verbose
  ```
  
  Current suite: 40 tests across 4 files (token, oracle, core, zapper),
  including a 256-run fuzz test proving the solvency invariant
  (basket holdings ≥ outstanding $NEXAI owed) holds under randomized
  deposit/redeem sequences with deliberately non-clean weight ratios.

  ## Deployment (Testnet)
  
  Scripts in `script/`, run in this order:
  
  1. `Deploy.s.sol` — deploys `NexaiIndexToken`, `NexaiCore`, mock
     WETH/USDC, grants `CORE_ROLE`, configures the basket
  2. `DeployUsdt.s.sol` — deploys mock USDT (no reliable real testnet
     USDT exists)
  3. `SeedPool.s.sol` — creates/seeds the WETH/USDC Uniswap V3 pool
  4. `SeedUsdtPools.s.sol` — creates/seeds USDT/WETH and USDT/USDC pools
     (fill in the USDT address from step 2 first)
  5. `DeployZapper.s.sol` — deploys `NexaiZapper` (fill in the USDT
     address first)
  6. `TopUpPools.s.sol` — adds liquidity to existing pools if depleted
     from testing (thin, self-seeded pools drain with heavy use — see
     "Known Limitations")
  
  Always dry-run first (omit `--broadcast`), then broadcast with
  `--slow` (see Known Limitations re: EIP-7702 in-flight tx limit).
  
  ```bash
  forge script script/Deploy.s.sol --rpc-url $BASE_SEPOLIA_RPC_URL --private-key $PRIVATE_KEY --broadcast --slow
  ```
  
  After deploying, verify each contract:
  ```bash
  forge verify-contract <ADDRESS> src/<Contract>.sol:<Contract> --chain base-sepolia --etherscan-api-key <KEY>
  ```

  ## Known Limitations (Testnet-Specific)
  
  - **Thin, self-seeded pools drain from heavy testing** — real trades
    against small pools cause genuine price impact, not a bug. Use
    `TopUpPools.s.sol` if swaps start failing with "STF" errors.
  - **RedStone integration is untested beyond a stubbed mock** — real
    signed-payload verification needs testnet integration testing before
    being trusted, not just unit tests with a bypassed mock.
  - **Zapper cost estimates in the frontend use a live Uniswap Quoter
    read**, which stays accurate but means quotes can shift between
    page-load and transaction submission — buffered at 5% currently.
  - **`updateBasket()` doesn't fund Core** — adding a new token via
    basket update alone breaks redemption for existing holders until
    Core is separately funded with it. See
    `/docs/NEXAI_Multisig_Timelock_Design.pdf` section 4.1 for the
    planned fix (funding check + pre-check view function).
  
  ## Roadmap / Design Docs
  
  - `/docs/NEXAI_Multisig_Timelock_Design.pdf` — admin structure
    (2-of-3 multisig, 48h timelock, 1-of-3 guardian pause), basket
    funding-check design
  - `/docs/M2_Basket_Scaling_Design.pdf` — scaling to the real 20-token
    basket: oracle setup, gas considerations, zapper generalization,
    dutch-auction rebalancing
