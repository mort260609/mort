  # Life OS — Product & System Design
  
  Personal RPG-style life management system. Real actions → XP, levels, ranks, stats, streaks — delivered as a true native "System"-style app with full-screen quest alerts, like Solo Leveling.
  
  ---
  
  ## 1. Product Architecture
  
  **Pattern:** React Native (Expo) native app + Supabase backend. No website — this is a phone-first app you install, with real OS-level push notifications and full-screen in-app "System" alert overlays.
  
  ```
  ┌─────────────────────────────────────────────┐
  │  Expo App (React Native, iOS + Android)      │
  │  ├─ app/(auth)/            → login/signup    │
  │  ├─ app/(tabs)/                              │
  │  │   ├─ dashboard/         → HUD overview    │
  │  │   ├─ quests/daily/      → daily quests    │
  │  │   ├─ quests/main/       → main quests     │
  │  │   ├─ stats/             → stat breakdown  │
  │  │   └─ analytics/         → charts/history  │
  │  ├─ components/                              │
  │  │   ├─ hud/               → HUD widgets     │
  │  │   └─ system-alert/      → full-screen     │
  │  │        "[System]" popup overlay           │
  │  ├─ lib/xp-engine/         → XP/level/rank   │
  │  │                            logic (pure)    │
  │  ├─ lib/notifications/     → Expo push        │
  │  │                            registration    │
  │  └─ lib/supabase/          → client helpers   │
  └─────────────────────────────────────────────┘
                │
                ▼
  ┌─────────────────────────────────────────────┐
  │  Supabase                                    │
  │  ├─ Postgres (schema below)                  │
  │  ├─ Row Level Security (per user_id)         │
  │  ├─ Auth (email/password or magic link)      │
  │  ├─ Edge Functions:                          │
  │  │   • daily streak rollover / quest reset   │
  │  │   • scheduled quest reminder pushes       │
  │  │   • level-up / milestone push triggers    │
  │  └─ Realtime (optional): live XP sync if you │
  │      ever add a second device                │
  └─────────────────────────────────────────────┘
                │
                ▼
  ┌─────────────────────────────────────────────┐
  │  Expo Push Notification Service              │
  │  → delivers native OS push notifications     │
  │    which the app intercepts and can render   │
  │    as a full-screen "System" alert on open   │
  └─────────────────────────────────────────────┘
  ```
  
  **Deployment:** EAS Build (Expo Application Services) compiles the native iOS/Android binaries. EAS Submit pushes to TestFlight / Play internal testing — you don't need to publish publicly, this can stay a personal app on your own devices via TestFlight/internal track indefinitely. Since you're Android-first, you can skip the Apple Developer Program ($99/yr) entirely for now — Android builds can be side-loaded or distributed via Play internal testing for free, and you can add iOS later only if you need it.
  
  **Key architectural decision:** all XP/level/rank math lives in a pure, framework-agnostic `lib/xp-engine` module — no DB calls inside it. The DB stores raw events (task completions), and the engine derives level/rank/stats from those events. This means you can recompute everything if you change the formula later, and it's trivially testable.
  
  **State flow:** Task completion → insert row in `quest_completions` → recalculates total XP, stat totals, streak → updates `profiles` cache columns → UI reflects new state optimistically, then confirms. Certain events (level-up, main quest completion, missed daily quest by evening) also fire a push notification via an Edge Function.
  
  ---
  
  ## 2. Database Schema (Supabase/Postgres)
  
  ```sql
  -- Core identity
  profiles (
    id                uuid primary key references auth.users,
    username          text,
    avatar_url        text,
    total_xp          int default 0,        -- cached, derived from xp_log
    current_level     int default 1,        -- cached
    gcoin_balance     int default 0,
    current_streak    int default 0,
    best_streak       int default 0,
    active_title_id   uuid,                 -- references titles(id), nullable
    last_active_date  date,
    created_at        timestamptz default now()
  )
  
  -- Stat totals (one row per stat per user)
  stats (
    id           uuid primary key default gen_random_uuid(),
    user_id      uuid references profiles(id),
    stat_name    text check (stat_name in
                  ('health','wealth','knowledge','discipline','network')),
    points       int default 0,
    unique(user_id, stat_name)
  )
  
  -- Daily Quest definitions (recurring habit templates)
  daily_quests (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    title         text not null,
    description   text,
    quest_rank    text check (quest_rank in ('E','D','C','B','A','S','SS')) not null,
    xp_reward     int not null,       -- derived from quest_rank, editable
    gcoin_reward  int not null,       -- derived from quest_rank, editable
    stat_target   text,          -- which stat this feeds
    stat_amount   int default 1,
    verification_type text check (verification_type in
                  ('self_report','photo','link','timer','integration','location')) default 'self_report',
    verification_config jsonb,   -- e.g. { min_minutes: 20 } for timer,
                                  -- { lat, lng, radius_m } for location,
                                  -- { source: 'health_connect', metric: 'exercise_minutes', min: 20 } for integration
    active        boolean default true,
    sort_order    int default 0,
    created_at    timestamptz default now()
  )
  
  -- Daily Quest completion log (one row per day completed)
  quest_completions (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    quest_id      uuid references daily_quests(id),
    completed_on  date not null,
    xp_awarded    int not null,
    gcoin_awarded int not null default 0,
    verification_type text check (verification_type in
                  ('self_report','photo','link','timer','integration','location')) default 'self_report',
    proof_url     text,              -- photo/screenshot storage path or link
    verified      boolean default false,  -- true once proof/integration check passes
    verified_at   timestamptz,
    unique(quest_id, completed_on)   -- prevents double-completion same day
  )
  
  -- Main Quests (long-term goals)
  main_quests (
    id              uuid primary key default gen_random_uuid(),
    user_id         uuid references profiles(id),
    title           text not null,
    description     text,
    quest_rank      text check (quest_rank in ('S','SS')) default 'S',
    xp_reward       int not null,       -- awarded on full completion
    status          text check (status in ('active','completed','abandoned')) default 'active',
    progress_pct    numeric default 0,  -- 0-100, derived from milestones
    created_at      timestamptz default now(),
    completed_at    timestamptz
  )
  
  -- Milestones within a Main Quest
  milestones (
    id              uuid primary key default gen_random_uuid(),
    main_quest_id   uuid references main_quests(id),
    title           text not null,
    xp_reward       int default 0,
    completed       boolean default false,
    completed_at    timestamptz,
    sort_order      int default 0
  )
  
  -- Immutable XP transaction log (source of truth)
  xp_log (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    source_type   text check (source_type in ('daily_quest','milestone','main_quest')),
    source_id     uuid,             -- FK to quest/milestone (soft ref)
    amount        int not null,
    created_at    timestamptz default now()
  )
  
  -- Streak history (per calendar day, did they complete ≥1 quest)
  streak_days (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    day           date not null,
    quests_done   int default 0,
    unique(user_id, day)
  )
  
  -- Trait definitions (permanent passive achievements)
  traits (
    id              uuid primary key default gen_random_uuid(),
    name            text not null,           -- e.g. "Iron Body"
    description     text,
    rarity          text check (rarity in ('common','uncommon','rare','epic','legendary')),
    unlock_condition jsonb,                  -- e.g. { type: 'quest_count', quest_tag: 'workout', count: 100 }
    passive_bonus   jsonb                    -- e.g. { stat: 'health', flat_bonus: 5 }
  )
  
  user_traits (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    trait_id      uuid references traits(id),
    unlocked_at   timestamptz default now(),
    unique(user_id, trait_id)
  )
  
  -- Title definitions (accomplishment-based, cosmetic display)
  titles (
    id              uuid primary key default gen_random_uuid(),
    name            text not null,           -- e.g. "Empire Architect"
    unlock_condition jsonb
  )
  
  user_titles (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    title_id      uuid references titles(id),
    unlocked_at   timestamptz default now(),
    unique(user_id, title_id)
  )
  -- profiles.active_title_id points at the one currently displayed
  
  -- Achievement definitions, including hidden ones
  achievements (
    id              uuid primary key default gen_random_uuid(),
    name            text not null,
    description     text,
    hidden          boolean default false,   -- true = not shown until unlocked
    unlock_condition jsonb,                  -- e.g. { type: 'streak', days: 30, before_hour: 6 }
    xp_reward       int default 0,
    gcoin_reward    int default 0,
    title_reward_id uuid references titles(id)  -- optional: unlocking it also grants a title
  )
  
  user_achievements (
    id              uuid primary key default gen_random_uuid(),
    user_id         uuid references profiles(id),
    achievement_id  uuid references achievements(id),
    unlocked_at     timestamptz default now(),
    unique(user_id, achievement_id)
  )
  ```
  
  **Why an immutable `xp_log`:** `profiles.total_xp` is a cache for fast reads, but `xp_log` is the ground truth. Analytics ("XP this week," "most improved stat") just query/aggregate this table — no separate analytics tables needed.
  
  ---
  
  ## 3. User Flow
  
  ```
  Sign up / log in
        │
        ▼
  Onboarding (first login only)
    → set username
    → pick 3-6 starter Daily Quests from templates (or custom)
    → set 1-3 starter Main Quests
        │
        ▼
  Dashboard (home base)
    → see Level, XP bar, Rank, Streak, Stat radar
        │
        ├──► Daily Quests tab
        │     → tap quest → mark complete → XP toast + stat tick
        │     → animated level-up modal if threshold crossed
        │
        ├──► Main Quests tab
        │     → open quest → view milestones → check one off
        │     → progress bar fills → big XP reward on 100%
        │
        ├──► Stats tab
        │     → radar/bar chart of 5 stats
        │     → tap a stat → see which quests feed it
        │
        └──► Analytics tab
              → weekly XP graph, monthly completions,
                best streak, most-improved stat
  ```
  
  Daily reset: at local midnight, quest "completed" checkboxes clear for the new day (completion history stays in `quest_completions`). A cron/edge function updates `streak_days`, and increments/resets `current_streak`.
  
  ---
  
  ## 4. XP System Design
  
  Every action funnels into one function: `awardXP(userId, amount, source)`. This:
  1. Inserts into `xp_log`
  2. Increments `profiles.total_xp`
  3. Recomputes `current_level` from `total_xp` (see §5)
  4. Updates relevant `stats` row(s)
  5. Updates `streak_days` for today
  
  **Quest Rank system:** replaces the earlier simple "difficulty" concept — every quest (Daily or Main) has a rank from E to SS, and rank is what drives XP, G-coin, and stat growth together, not just XP alone:
  
  | Rank | XP | G-Coin | Stat Growth | Example |
  |---|---|---|---|---|
  | E | 10 | 2 | +1 | Make bed, drink water |
  | D | 25 | 5 | +1 | Morning prayer, read 10 pages |
  | C | 50 | 10 | +2 | Workout, journal entry |
  | B | 100 | 20 | +3 | Trading study session, content creation |
  | A | 250 | 50 | +5 | Build a product feature, major project task |
  | S | 500 | 100 | +8 | Pass prop firm challenge, secure new income source |
  | SS | 1000 | 200 | +12 | Major life achievement, relocation milestone |
  
  The G-coin and stat-growth columns are a suggested starting ratio (roughly 1/5 of XP for G-coin) — tune freely once you see how fast the shop economy moves in practice.
  
  **Main Quest milestones:** typically 50–200 XP each; the final completion of the whole Main Quest awards a large bonus (e.g. 500–2000 XP) on top of milestone totals — this makes big goals feel disproportionately rewarding, matching real-life stakes.
  
  **Buffs / XP Modifiers:** all bonus-XP mechanics (streak bonus, quest-triggered boosts, shop-bought boosts) run through one system instead of separate hacks. `awardXP` looks up your active buffs, sums their multipliers (capped, see below), and applies the total to the base XP before logging.
  
  ```sql
  buffs (
    id             uuid primary key default gen_random_uuid(),
    user_id        uuid references profiles(id),
    name           text not null,           -- e.g. "Focused Mind", "Streak Momentum"
    xp_multiplier  numeric not null,        -- 0.05 = +5%
    source_type    text check (source_type in ('quest_completion','streak','shop_purchase')),
    source_id      uuid,                    -- which quest/item granted it
    granted_at     timestamptz default now(),
    expires_at     timestamptz,             -- null = permanent
    active         boolean default true
  )
  ```
  
  Examples of how each source grants a buff:
  - **Quest-triggered:** a specific quest (e.g. an Epic-difficulty or Main Quest milestone) grants a named buff like *"[System] Buff acquired: Focused Mind (+5% XP, 24h)"* on completion — set `expires_at` for temporary, leave null for a permanent unlock (e.g. finishing a Main Quest could permanently buff quests tied to the same stat).
  - **Streak:** the existing streak bonus becomes just another row, recalculated daily: `+1% per streak day, capped at +30%`.
  - **Shop-bought:** a G-coin shop item ("Mana Elixir — +10% XP, 12h") inserts a temporary buff on purchase — a genuinely useful thing to spend G-coin on besides cosmetics.
  
  **Stacking rule:** sum all active multipliers, but cap the total (e.g. +50% max) so buffs stay a nice bonus rather than letting XP snowball out of control. `finalXP = baseXP * (1 + min(sum(activeMultipliers), 0.5))`.

  ## 5. Level Progression System
  
  Use an escalating curve so early levels feel fast (dopamine hook) and later levels feel earned.
  
  **Formula (XP required to go from level `n` to `n+1`):**
  
  ```
  xpForLevel(n) = round(100 * n^1.5)
  ```
  
  | Level | XP for this level | Cumulative XP |
  |---|---|---|
  | 1→2 | 100 | 100 |
  | 2→3 | 283 | 383 |
  | 3→4 | 520 | 903 |
  | 5→6 | 1,118 | ~2,850 |
  | 10→11 | 3,162 | ~11,600 |
  | 20→21 | 8,944 | ~57,900 |
  
  **Deriving current level from `total_xp`:** precompute a cumulative-XP lookup table (or invert the formula) in `lib/xp-engine/level.ts`. Since it's a pure function of `total_xp`, level is always derivable — never stored as the source of truth, only cached.
  
  ```ts
  function getLevel(totalXp: number): { level: number; xpIntoLevel: number; xpForNext: number } {
    let level = 1;
    let cumulative = 0;
    while (true) {
      const need = Math.round(100 * Math.pow(level, 1.5));
      if (cumulative + need > totalXp) {
        return { level, xpIntoLevel: totalXp - cumulative, xpForNext: need };
      }
      cumulative += need;
      level++;
    }
  }
  ```
  
  This is cheap enough to run on every render for realistic level ranges (under a few hundred).

  | Apprentice | 5–9 |

  ## 7. Streak System
  
  Three tracked windows, all derived from `streak_days`:
  
  - **Daily streak:** consecutive calendar days with ≥1 quest completed. Breaks reset to 0 (with a "streak freeze" grace item as a nice-to-have later).
  - **Weekly streak:** consecutive weeks where ≥N quests completed (define N, e.g. 5) — good for forgiving the occasional missed day.
  - **Monthly streak:** consecutive months hitting a completion-rate threshold (e.g. 70% of days active).
  
  `best_streak` on `profiles` is just `max(current_streak)` ever seen — update it whenever `current_streak` increases.

  ## 8. Quest Verification System
  
  Self-report ("just tap complete") is fine for low-stakes habits, but for anything you're likely to fudge when unmotivated, the quest itself should carry a verification method — this is what makes the XP feel earned instead of arbitrary.
  
  **Verification tiers** (weakest to strongest, stored per-quest as `verification_type`):
  
  | Tier | How it works | Best for |
  |---|---|---|
  | `self_report` | Tap complete, no proof | Prayer, meditation, low-stakes habits |
  | `photo` | Attach a photo on completion, stored in Supabase Storage | Workout (gym pic), reading (progress photo) |
  | `link` | Paste a URL/screenshot as proof | Posted content, job application confirmation |
  | `timer` | Must run an in-app timer for a minimum duration before "complete" unlocks | Study sessions, focused trading study |
  | `integration` | Pulled automatically from a connected data source | Workout minutes/steps via Android Health Connect, screen-time via device usage APIs |
  | `location` | Must be within a saved GPS radius | Gym check-in, library/study spot |
  
  **Suggested mapping for your example quests:**
  
  - Workout → `integration` (Health Connect exercise minutes) or `photo` if you'd rather not connect health data
  - Morning prayer → `self_report` (some things don't need policing)
  - Study trading → `timer` (minimum focused minutes)
  - Post content → `link` (paste the post URL)
  - Read a book → `photo` (progress shot) or `timer`
  - Apply for jobs → `link` or `photo` (confirmation email/screenshot)
  
  **How it plugs into the schema:** `daily_quests.verification_type` + `verification_config` (jsonb) define the requirement when you create the quest (e.g. `{ min_minutes: 20 }` for timer, `{ lat, lng, radius_m }` for location). `quest_completions.proof_url` / `verified` / `verified_at` capture what was actually submitted. XP is awarded immediately for `self_report`, but for the stronger tiers the completion can be marked `verified: false` until the proof/integration check passes — so the XP toast still feels instant, while the "[System]" alert log can honestly show which completions were verified vs. self-reported if you ever want to see your own integrity stats.
  
  **Recommendation:** don't verify everything on day one — start most quests as `self_report` in Phase 1, then upgrade your highest-stakes quests (workout, study) to `photo`/`timer`/`integration` once the core loop is working. This is naturally a Phase 3–4 addition, not a blocker for MVP.

  ## 9. Stats System

Stats have no cap and no separate "level" — they're a lifetime accumulation, visualized as a radar chart on the dashboard so you can see which parts of life are getting neglected at a glance.
  
  Optional future layer: derive a per-stat "level" the same way you derive character level, using a smaller XP curve, so stats feel more RPG-like individually (e.g. "Discipline Lv. 7").

  ## 11. AI Quest Master (Goal Planner) — Designed for Later Upgrade
  
  **Role clarity, since this changes how the whole app should feel:** the AI Quest Master isn't a side feature bolted onto an otherwise static app — it *is* the System. You're the **Architect** (you design the goals, own the app, ultimately decide what gets accepted) and the **Player** (you're the one out in the real world executing quests). The System is the third role — the admin — and once the AI Quest Master exists, it should be the single voice behind every "[System]" alert (§12), not just the goal-planning screen. That means:
  
  - Level-up, rank-up, and quest-issued alerts should eventually be phrased *by* the AI in-character as the System, not pulled from a static template
  - Missed-quest/punishment alerts get their bite from the same voice — consistent, a little cold, always addressing you as the Player
  - The Quest Master's goal-breakdown feature (below) is just the most functional expression of that same admin role — the System doesn't just talk strict, it actually plans and enforces
  
  Practically, this means Phase 1's static alert templates can ship first (so the core loop works without needing an LLM call on every event), and Phase 6 doesn't just add a planning screen — it upgrades the *voice* of every existing alert to route through the AI instead of a template, once the Quest Master exists.
  
  Fully feasible, and it slots in cleanly without touching the core schema — it just becomes another way rows get created in `main_quests`, `milestones`, and `daily_quests`.
  
  **How it works:**
  
  ```
  You type/speak a goal
    ("I want to pass a prop firm challenge")
          │
          ▼
  Edge Function calls an LLM (e.g. Claude via
  the Anthropic API) with the goal + your
  current stats/quest list as context, and a
  strict instruction to return ONLY structured
  JSON matching your schema:
    {
      main_quest: { title, description, xp_reward },
      milestones: [ { title, xp_reward }, ... ],
      suggested_daily_quests: [
        { title, xp_reward, difficulty, stat_target }, ...
      ]
    }
          │
          ▼
  App shows you the proposed plan as a review
  screen — edit/remove/accept items individually
          │
          ▼
  On accept, rows are inserted into main_quests /
  milestones / daily_quests exactly like if you'd
  created them by hand
  ```
  
  **Why this design works well for you specifically:**
  - The AI never writes directly to your DB — it proposes, you approve. Keeps you in control and avoids junk quests polluting your list.
  - The API call happens server-side (Supabase Edge Function), so your API key never sits in the mobile app bundle.
  - Because it outputs the *same shape* your manual quest-creation forms already produce, this is an additive feature — nothing in Phases 0–5 needs to change to support it later. You can genuinely ship the manual version first and bolt this on afterward.
  - Optional nice extension once this exists: a periodic "Quest Master check-in" push notification — the AI reviews your recent completions/stats and proposes a new quest, rather than you always having to initiate it.
  
  **What it needs when you build it:** an LLM API key (Anthropic API works well for this — strong at structured JSON output and reasoning about multi-step plans), one Edge Function endpoint, and a review/accept UI screen. No schema changes required.

  ## 12. "System" Alert / Notification Design
  
  This is the signature feel of the app — the mechanic that makes it feel alive instead of like a checklist.
  
  **Two layers:**
  
  1. **OS push notifications** (via Expo Notifications + Supabase Edge Function cron): fire even when the app is closed. Examples:
     - Morning: *"[System] Daily Quests have been issued."*
     - Evening, quest still incomplete: *"[System] Warning: 'Workout' quest incomplete. Time remaining: 3h."*
     - Streak at risk: *"[System] Your streak will end in 2 hours."*
  2. **In-app full-screen "System" overlay** (rendered from `components/system-alert/`): triggered when the app is opened after a push, or immediately after a significant in-app event. Mimics the Solo Leveling blue-panel "System" window — dark translucent overlay, gold/white monospace-ish text, typewriter reveal animation, single "Accept" tap to dismiss. Used for:
     - New daily quests issued (start of day)
     - Level up ("[System] You have reached Level 12.")
     - Rank up ("[System] Rank promoted: Scout → Warrior.")
     - Main Quest milestone or full completion
     - Missed quest / streak broken (a harsher, red-tinted variant — this is the "consequence" beat that makes it feel real)
  
  **Why the split:** push notifications are what make it feel like a system is watching your life even when the phone's locked; the full-screen overlay is what makes the moment of opening the app feel like a cutscene, not a status check.
  
  **Tech notes:** Expo Notifications handles device push tokens and OS-level delivery; a Supabase Edge Function on a cron schedule checks each user's incomplete quests/streak status daily and calls the Expo Push API. In-app overlays are just local state driven by a queue — any XP-engine event (level up, quest complete, streak broken) pushes an alert object onto the queue, and the overlay component renders the next one in line.

  ## 13. Smart Glasses Integration (Future) — Feasibility
  
  Realistic, and it splits into two tiers depending on how deep you want to go — worth knowing both now so the notification system (§11) is built in a way that doesn't block either path later.
  
  **Tier 1 — Notification mirroring (easy, works with most Android-compatible glasses today):**
  Most current Android smart glasses (e.g. Even Realities G1, Meta Ray-Ban Display, various Android XR-based glasses) work by mirroring your phone's notifications to a small HUD via Bluetooth, using a companion app. Since your app already fires real Android push/local notifications for "[System]" alerts, quest-issued messages, and level-ups, those notifications would **already show up on this class of glasses automatically** — no extra integration work needed, as long as the glasses' companion app is configured to mirror app notifications. This tier is essentially "free" once Phase 3 (notifications) is done.
  
  **Tier 2 — Native glasses app / custom HUD (harder, hardware-dependent):**
  Some platforms (notably Android XR, Google's glasses/headset platform announced for 2025–2026 devices, and vendor-specific SDKs like Even Realities' own SDK) let you build a dedicated glasses-side app or display card — e.g. a persistent mini-HUD showing your XP bar, current quest, or streak directly on the lens, not just a notification popup. This requires:
  - Picking a specific glasses platform/SDK once you own the hardware (SDKs vary a lot and aren't interchangeable)
  - A lightweight "glanceable" UI package separate from the phone app, reading the same Supabase backend
  - No changes to the core schema or XP engine — this tier is purely a new display client on top of what you're already building
  
  **Recommendation:** don't design for a specific glasses SDK now — hardware and SDKs in this space are still shifting fast. Just make sure (as already planned) that meaningful events flow through real device notifications, and treat glasses as a future "second client" reading the same backend, the same way the AI Quest Master is a future add-on.

  ## 14. Avatar System
  
  **Onboarding:** you enter height/weight/body metrics manually (a photo alone can't measure these accurately — no reference scale to calibrate against). You also upload a full-body photo purely for art generation, and set your calisthenics goal.
  
  **Avatar generation:** an Edge Function sends your photo to an image-generation API (img2img/style-transfer) with a fantasy-system art style prompt, producing one stylized base avatar. This becomes your permanent character art — not regenerated from scratch later, since repeated AI regeneration drifts and won't look like "the same character" level to level.
  
  **Progression:** instead of regenerating the whole avatar, progress is shown through **layered overlays** on the same base art — armor pieces, an aura/glow effect, and shading changes — unlocked at level/stat thresholds (e.g. Discipline/Health stat milestones tied to your calisthenics goal). This is cheap, fast, and visually consistent, and mirrors exactly how manhwa "system" characters visibly level up without becoming a different person each panel.
  
  **Schema:**
  ```sql
  avatars (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    source_photo_url  text,      -- original upload, private storage
    base_avatar_url   text,      -- generated stylized art
    height_cm     numeric,
    weight_kg     numeric,
    goal_type     text,          -- e.g. 'calisthenics'
    current_tier  int default 1, -- drives which overlays render
    updated_at    timestamptz default now()
  )
  
  avatar_overlays (
    id            uuid primary key default gen_random_uuid(),
    tier          int not null,           -- unlock threshold
    overlay_type  text,                   -- 'armor' | 'aura' | 'shading'
    asset_url     text
  )
  ```

  ```sql
  gcoin_log (               -- immutable ledger, same pattern as xp_log
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    amount        int not null,           -- positive (earn) or negative (spend)
    source_type   text,
    source_id     uuid,

  )
  
  shop_items (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),   -- you define your own shop
    title         text not null,
    description   text,
    cost_gcoin    int not null,

  purchases (
    id            uuid primary key default gen_random_uuid(),

    purchased_at  timestamptz default now()
  )
  ```
  
  Since it's your own personal system, you define the shop items yourself — cosmetic ones (avatar overlays, HUD themes/titles) and real-world ones (e.g. "300 G-coin = guilt-free takeout night," "1000 G-coin = new game"). The AI Quest Master (§11) could later auto-suggest reward items scaled to a goal's difficulty, but that's an easy bolt-on, not needed for MVP.

  ## 18. Punishment / Discipline System
  
  Two real implementation paths on Android — worth being upfront about the tradeoff:
  
  | Approach | What it can actually do | Setup cost | Risk |
  |---|---|---|---|
  | **Accessibility-Service app blocker** (recommended) | Hard-blocks specific apps (social media, games, whatever you name) until a quest is completed. Emergency calling/essentials always stay accessible. | Just a permission grant, no special provisioning | Low — worst case you disable the permission |
  | **Device Owner mode** (full lockout) | Full-device lock, like enterprise MDM/parental controls | Must be provisioned via ADB *before* first Google sign-in on the device | High — misconfigure it and you may need a full factory reset to regain access |
  
  **Recommendation:** build the Accessibility-Service blocker. It's genuinely strict (you cannot open the blocked apps, no workaround from inside them) without carrying the risk of locking yourself out of your own phone over a missed habit.
  
  **Punishment tiers (config, not schema — logic lives in the Edge Function that runs the daily rollover):**
  - Missed 1 daily quest → blocked apps (your choice list) until first quest of the next day is completed
  - Broken streak → longer block window + G-coin penalty + a stat "decay" tick
  - Missed Main Quest milestone deadline (if you set one) → System sends a harsher red-tinted alert, no XP penalty by default (main quests are long-term, punishing missed deadlines too hard tends to just make people quit instead of restart)
  
  ```sql
  punishment_rules (
    id             uuid primary key default gen_random_uuid(),
    user_id        uuid references profiles(id),
    trigger_type   text check (trigger_type in ('missed_quest','broken_streak','missed_milestone')),
    blocked_apps   text[],       -- package names to block
    block_duration_hours int,
    gcoin_penalty  int default 0,
    active         boolean default true
  )
  ```

  ## 19. Trait System
  
  Permanent passive achievements earned through *consistent* behavior over time — distinct from single quest completions, and the mechanism that rewards showing up repeatedly rather than any one good day.
  
  Traits are defined once (`traits` table, §2) with an `unlock_condition` (e.g. "100 quests tagged workout completed"), a rarity tier (common → legendary), and an optional `passive_bonus` (a small permanent stat or XP effect — e.g. "Iron Body" could grant +5 Health permanently). A daily/nightly Edge Function job checks unlock conditions against your completion history and inserts into `user_traits` when met, firing a "[System] Trait Unlocked" alert.
  
  Example progression:
  - 100 Workouts → **Iron Body** (uncommon)
  - 100 Trading Sessions → **Market Student** (uncommon)
  - 30 Consecutive Days Posting Content → **Consistent Creator** (rare)
  - 365 Days Active → **Unbroken Will** (legendary)
  
  Traits are visible on the Traits & Titles screen (§10) — this is the "long-term identity" layer that Level/Rank alone doesn't capture, since a level resets in feel every time you level up again, but a Trait, once earned, is permanent proof of who you've become.

  ## 20. Titles System
  
  Titles are separate from Rank: **Rank measures how far you've progressed overall; Titles represent specific things you've accomplished.** You can only display one at a time, so equipping one is itself a small identity statement.
  
  Defined in `titles`, unlocked into `user_titles` the same way traits are — either standalone (via an `unlock_condition`) or as a reward attached to a Hidden Achievement (§21). `profiles.active_title_id` holds the currently equipped one, shown right under Rank on the Status Window:
  
  ```
  Rank: Elite
  Title: Empire Architect
  ```
  
  Example titles: **The Disciplined**, **The Builder**, **The Scholar**, **The Trader**, **The Survivor**, **Empire Architect** — you can define your own set matching your actual goals (trading, relocation, building this app itself) rather than generic RPG flavor.

  ## 21. Hidden Achievements System
  
  Achievements with `hidden: true` (§2 schema) simply aren't rendered anywhere in the UI — not in a locked/silhouette state, not listed at all — until their `unlock_condition` is met. This is what makes unlocking one feel like a genuine surprise rather than a checklist you were quietly grinding toward.
  
  Checked by the same nightly Edge Function job as Traits, against whatever data the condition needs (streaks, cumulative quest counts, specific stat thresholds, even something like "logged before 6 AM for 30 days" pulled from `quest_completions.completed_on` timestamps). On unlock: a distinct, more dramatic "[System]" alert variant (longer reveal animation, unique sound) plus XP/G-coin/title rewards as configured on the achievement row.
  
  Example hidden achievements: Wake before 6AM for 30 days, complete 100 total quests, earn your first $100 online, run 100km total, reach Level 50 — a mix of behavioral consistency and milestone thresholds.

  ## 22. Progression Philosophy
  
  Worth stating explicitly since it should guide every feature decision going forward: **this app is not a checklist — it's identity transformation.** You're not tracking tasks, you're becoming a different version of yourself through repetition, and every system above exists to make that legible:
  
  ```
  Action → Quest Completion → XP Gain → Level Increase
         → Stat Growth → Traits Unlocked → Titles Earned
         → Identity Evolution
  ```
  
  When evaluating any future feature (including whatever comes after Phase 7), the test is: does this reinforce that loop, or is it just another number on a screen? XP/Level answer "how much have I done." Stats answer "in what direction." Traits and Titles answer "who have I become." All three matter; none of them replace the others.

  # THE SYSTEM — Product & Technical Specification
  
  Personal RPG-style life management system. Real actions → XP, levels, ranks, stats, streaks — delivered as a true native "System"-style app with full-screen quest alerts, like Solo Leveling.
  
  **Codebase note:** this document is the full spec for handoff to an implementation agent (Antigravity/Claude Code). §2 (schema) and stat/rank decisions in §4–6 are load-bearing — get these right before writing code. Everything from §14 onward (Avatar, G-Coin, Traits, Titles, AI Quest Master, Smart Glasses) is designed to be additive: safe to scaffold as empty schema now, safe to leave unbuilt until later phases per §23.
  
  ---
  
  ## 1. Product Architecture
  
  **Pattern:** React Native (Expo) native app + Supabase backend. No website — this is a phone-first app you install, with real OS-level push notifications and full-screen in-app "System" alert overlays.
  
  ```
  ┌─────────────────────────────────────────────┐
  │  Expo App (React Native, iOS + Android)      │
  │  ├─ app/(auth)/            → login/signup    │
  │  ├─ app/(tabs)/                              │
  │  │   ├─ dashboard/         → HUD overview    │
  │  │   ├─ quests/daily/      → daily quests    │
  │  │   ├─ quests/main/       → main quests     │
  │  │   ├─ stats/             → stat breakdown  │
  │  │   └─ analytics/         → charts/history  │
  │  ├─ components/                              │
  │  │   ├─ hud/               → HUD widgets     │
  │  │   └─ system-alert/      → full-screen     │
  │  │        "[System]" popup overlay           │
  │  ├─ lib/xp-engine/         → XP/level/rank   │
  │  │                            logic (pure)    │
  │  ├─ lib/notifications/     → Expo push        │
  │  │                            registration    │
  │  └─ lib/supabase/          → client helpers   │
  └─────────────────────────────────────────────┘
                │
                ▼
  ┌─────────────────────────────────────────────┐
  │  Supabase                                    │
  │  ├─ Postgres (schema below)                  │
  │  ├─ Row Level Security (per user_id)         │
  │  ├─ Auth (email/password or magic link)      │
  │  ├─ Edge Functions:                          │
  │  │   • daily streak rollover / quest reset   │
  │  │   • scheduled quest reminder pushes       │
  │  │   • level-up / milestone push triggers    │
  │  └─ Realtime (optional): live XP sync if you │
  │      ever add a second device                │
  └─────────────────────────────────────────────┘
                │
                ▼
  ┌─────────────────────────────────────────────┐
  │  Expo Push Notification Service              │
  │  → delivers native OS push notifications     │
  │    which the app intercepts and can render   │
  │    as a full-screen "System" alert on open   │
  └─────────────────────────────────────────────┘
  ```
  
  **Deployment:** EAS Build (Expo Application Services) compiles the native iOS/Android binaries. EAS Submit pushes to TestFlight / Play internal testing — you don't need to publish publicly, this can stay a personal app on your own devices via TestFlight/internal track indefinitely. Since you're Android-first, you can skip the Apple Developer Program ($99/yr) entirely for now — Android builds can be side-loaded or distributed via Play internal testing for free, and you can add iOS later only if you need it.
  
  **Key architectural decision:** all XP/level/rank math lives in a pure, framework-agnostic `lib/xp-engine` module — no DB calls inside it. The DB stores raw events (task completions), and the engine derives level/rank/stats from those events. This means you can recompute everything if you change the formula later, and it's trivially testable.
  
  **State flow:** Task completion → insert row in `quest_completions` → recalculates total XP, stat totals, streak → updates `profiles` cache columns → UI reflects new state optimistically, then confirms. Certain events (level-up, main quest completion, missed daily quest by evening) also fire a push notification via an Edge Function.

  ## 2. Database Schema (Supabase/Postgres)
  
  ```sql
  -- Core identity
  profiles (
    id                uuid primary key references auth.users,
    username          text,
    avatar_url        text,
    total_xp          int default 0,        -- cached, derived from xp_log
    current_level     int default 1,        -- cached
    gcoin_balance     int default 0,
    current_streak    int default 0,
    best_streak       int default 0,
    active_title_id   uuid,                 -- references titles(id), nullable
    last_active_date  date,
    created_at        timestamptz default now()
  )
  
  -- Stat totals (one row per stat per user)
  stats (
    id           uuid primary key default gen_random_uuid(),
    user_id      uuid references profiles(id),
    stat_name    text check (stat_name in
                  ('health','wealth','knowledge','discipline','network')),
    points       int default 0,
    unique(user_id, stat_name)
  )
  
  -- Daily Quest definitions (recurring habit templates)
  daily_quests (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    title         text not null,
    description   text,
    quest_rank    text check (quest_rank in ('E','D','C','B','A','S','SS')) not null,
    xp_reward     int not null,       -- derived from quest_rank, editable
    gcoin_reward  int not null,       -- derived from quest_rank, editable
    stat_target   text,          -- which stat this feeds
    stat_amount   int default 1,
    verification_type text check (verification_type in
                  ('self_report','photo','link','timer','integration','location')) default 'self_report',
    verification_config jsonb,   -- e.g. { min_minutes: 20 } for timer,
                                  -- { lat, lng, radius_m } for location,
                                  -- { source: 'health_connect', metric: 'exercise_minutes', min: 20 } for integration
    active        boolean default true,
    sort_order    int default 0,
    created_at    timestamptz default now()
  )
  
  -- Daily Quest completion log (one row per day completed)
  quest_completions (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    quest_id      uuid references daily_quests(id),
    completed_on  date not null,
    xp_awarded    int not null,
    gcoin_awarded int not null default 0,
    verification_type text check (verification_type in
                  ('self_report','photo','link','timer','integration','location')) default 'self_report',
    proof_url     text,              -- photo/screenshot storage path or link
    verified      boolean default false,  -- true once proof/integration check passes
    verified_at   timestamptz,
    unique(quest_id, completed_on)   -- prevents double-completion same day
  )
  
  -- Main Quests (long-term goals)
  main_quests (
    id              uuid primary key default gen_random_uuid(),
    user_id         uuid references profiles(id),
    title           text not null,
    description     text,
    quest_rank      text check (quest_rank in ('S','SS')) default 'S',
    xp_reward       int not null,       -- awarded on full completion
    status          text check (status in ('active','completed','abandoned')) default 'active',
    progress_pct    numeric default 0,  -- 0-100, derived from milestones
    created_at      timestamptz default now(),
    completed_at    timestamptz
  )
  
  -- Milestones within a Main Quest
  milestones (
    id              uuid primary key default gen_random_uuid(),
    main_quest_id   uuid references main_quests(id),
    title           text not null,
    xp_reward       int default 0,
    completed       boolean default false,
    completed_at    timestamptz,
    sort_order      int default 0
  )
  
  -- Immutable XP transaction log (source of truth)
  xp_log (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    source_type   text check (source_type in ('daily_quest','milestone','main_quest')),
    source_id     uuid,             -- FK to quest/milestone (soft ref)
    amount        int not null,
    created_at    timestamptz default now()
  )
  
  -- Streak history (per calendar day, did they complete ≥1 quest)
  streak_days (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    day           date not null,
    quests_done   int default 0,
    unique(user_id, day)
  )
  
  -- Trait definitions (permanent passive achievements)
  traits (
    id              uuid primary key default gen_random_uuid(),
    name            text not null,           -- e.g. "Iron Body"
    description     text,
    rarity          text check (rarity in ('common','uncommon','rare','epic','legendary')),
    unlock_condition jsonb,                  -- e.g. { type: 'quest_count', quest_tag: 'workout', count: 100 }
    passive_bonus   jsonb                    -- e.g. { stat: 'health', flat_bonus: 5 }
  )
  
  user_traits (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    trait_id      uuid references traits(id),
    unlocked_at   timestamptz default now(),
    unique(user_id, trait_id)
  )
  
  -- Title definitions (accomplishment-based, cosmetic display)
  titles (
    id              uuid primary key default gen_random_uuid(),
    name            text not null,           -- e.g. "Empire Architect"
    unlock_condition jsonb
  )
  
  user_titles (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    title_id      uuid references titles(id),
    unlocked_at   timestamptz default now(),
    unique(user_id, title_id)
  )
  -- profiles.active_title_id points at the one currently displayed
  
  -- Achievement definitions, including hidden ones
  achievements (
    id              uuid primary key default gen_random_uuid(),
    name            text not null,
    description     text,
    hidden          boolean default false,   -- true = not shown until unlocked
    unlock_condition jsonb,                  -- e.g. { type: 'streak', days: 30, before_hour: 6 }
    xp_reward       int default 0,
    gcoin_reward    int default 0,
    title_reward_id uuid references titles(id)  -- optional: unlocking it also grants a title
  )
  
  user_achievements (
    id              uuid primary key default gen_random_uuid(),
    user_id         uuid references profiles(id),
    achievement_id  uuid references achievements(id),
    unlocked_at     timestamptz default now(),
    unique(user_id, achievement_id)
  )
  ```
  
  **Why an immutable `xp_log`:** `profiles.total_xp` is a cache for fast reads, but `xp_log` is the ground truth. Analytics ("XP this week," "most improved stat") just query/aggregate this table — no separate analytics tables needed.

  ## 4. XP System Design
  
  Every action funnels into one function: `awardXP(userId, amount, source)`. This:
  1. Inserts into `xp_log`
  2. Increments `profiles.total_xp`
  3. Recomputes `current_level` from `total_xp` (see §5)
  4. Updates relevant `stats` row(s)
  5. Updates `streak_days` for today
  
  **Quest Rank system:** replaces the earlier simple "difficulty" concept — every quest (Daily or Main) has a rank from E to SS, and rank is what drives XP, G-coin, and stat growth together, not just XP alone:
  
  | Rank | XP | G-Coin | Stat Growth | Example |
  |---|---|---|---|---|
  | E | 10 | 2 | +1 | Make bed, drink water |
  | D | 25 | 5 | +1 | Morning prayer, read 10 pages |
  | C | 50 | 10 | +2 | Workout, journal entry |
  | B | 100 | 20 | +3 | Trading study session, content creation |
  | A | 250 | 50 | +5 | Build a product feature, major project task |
  | S | 500 | 100 | +8 | Pass prop firm challenge, secure new income source |
  | SS | 1000 | 200 | +12 | Major life achievement, relocation milestone |
  
  The G-coin and stat-growth columns are a suggested starting ratio (roughly 1/5 of XP for G-coin) — tune freely once you see how fast the shop economy moves in practice.
  
  **Main Quest milestones:** typically 50–200 XP each; the final completion of the whole Main Quest awards a large bonus (e.g. 500–2000 XP) on top of milestone totals — this makes big goals feel disproportionately rewarding, matching real-life stakes.
  
  **Buffs / XP Modifiers:** all bonus-XP mechanics (streak bonus, quest-triggered boosts, shop-bought boosts) run through one system instead of separate hacks. `awardXP` looks up your active buffs, sums their multipliers (capped, see below), and applies the total to the base XP before logging.
  
  ```sql
  buffs (
    id             uuid primary key default gen_random_uuid(),
    user_id        uuid references profiles(id),
    name           text not null,           -- e.g. "Focused Mind", "Streak Momentum"
    xp_multiplier  numeric not null,        -- 0.05 = +5%
    source_type    text check (source_type in ('quest_completion','streak','shop_purchase')),
    source_id      uuid,                    -- which quest/item granted it
    granted_at     timestamptz default now(),
    expires_at     timestamptz,             -- null = permanent
    active         boolean default true
  )
  ```
  
  Examples of how each source grants a buff:
  - **Quest-triggered:** a specific quest (e.g. an Epic-difficulty or Main Quest milestone) grants a named buff like *"[System] Buff acquired: Focused Mind (+5% XP, 24h)"* on completion — set `expires_at` for temporary, leave null for a permanent unlock (e.g. finishing a Main Quest could permanently buff quests tied to the same stat).
  - **Streak:** the existing streak bonus becomes just another row, recalculated daily: `+1% per streak day, capped at +30%`.
  - **Shop-bought:** a G-coin shop item ("Mana Elixir — +10% XP, 12h") inserts a temporary buff on purchase — a genuinely useful thing to spend G-coin on besides cosmetics.
  
  **Stacking rule:** sum all active multipliers, but cap the total (e.g. +50% max) so buffs stay a nice bonus rather than letting XP snowball out of control. `finalXP = baseXP * (1 + min(sum(activeMultipliers), 0.5))`.

  ## 5. Level Progression System
  
  Use an escalating curve so early levels feel fast (dopamine hook) and later levels feel earned.
  
  **Formula (XP required to go from level `n` to `n+1`):**
  
  ```
  xpForLevel(n) = round(100 * n^1.5)
  ```
  
  | Level | XP for this level | Cumulative XP |
  |---|---|---|
  | 1→2 | 100 | 100 |
  | 2→3 | 283 | 383 |
  | 3→4 | 520 | 903 |
  | 5→6 | 1,118 | ~2,850 |
  | 10→11 | 3,162 | ~11,600 |
  | 20→21 | 8,944 | ~57,900 |
  
  **Deriving current level from `total_xp`:** precompute a cumulative-XP lookup table (or invert the formula) in `lib/xp-engine/level.ts`. Since it's a pure function of `total_xp`, level is always derivable — never stored as the source of truth, only cached.
  
  ```ts
  function getLevel(totalXp: number): { level: number; xpIntoLevel: number; xpForNext: number } {
    let level = 1;
    let cumulative = 0;
    while (true) {
      const need = Math.round(100 * Math.pow(level, 1.5));
      if (cumulative + need > totalXp) {
        return { level, xpIntoLevel: totalXp - cumulative, xpForNext: need };
      }
      cumulative += need;
      level++;
    }
  }
  ```
  
  This is cheap enough to run on every render for realistic level ranges (under a few hundred).

  ## 6. Rank System
  
  Ranks are level-range bands — cosmetic titles layered on top of levels for extra flavor.
  
  | Rank | Level Range |
  |---|---|
  | Novice | 1–4 |
  | Apprentice | 5–9 |
  | Scout | 10–14 |
  | Warrior | 15–24 |
  | Elite | 25–34 |
  | Empire Builder | 35–49 |
  | Legend | 50+ |
  
  Store as a static config array `[{ name, minLevel }]` — derive rank the same way as level, no DB table needed unless you want per-rank unlockable perks/badges later.

  ## 7. Streak System
  
  Three tracked windows, all derived from `streak_days`:
  
  - **Daily streak:** consecutive calendar days with ≥1 quest completed. Breaks reset to 0 (with a "streak freeze" grace item as a nice-to-have later).
  - **Weekly streak:** consecutive weeks where ≥N quests completed (define N, e.g. 5) — good for forgiving the occasional missed day.
  - **Monthly streak:** consecutive months hitting a completion-rate threshold (e.g. 70% of days active).
  
  `best_streak` on `profiles` is just `max(current_streak)` ever seen — update it whenever `current_streak` increases.

  ## 8. Quest Verification System
  
  Self-report ("just tap complete") is fine for low-stakes habits, but for anything you're likely to fudge when unmotivated, the quest itself should carry a verification method — this is what makes the XP feel earned instead of arbitrary.
  
  **Verification tiers** (weakest to strongest, stored per-quest as `verification_type`):
  
  | Tier | How it works | Best for |
  |---|---|---|
  | `self_report` | Tap complete, no proof | Prayer, meditation, low-stakes habits |
  | `photo` | Attach a photo on completion, stored in Supabase Storage | Workout (gym pic), reading (progress photo) |
  | `link` | Paste a URL/screenshot as proof | Posted content, job application confirmation |
  | `timer` | Must run an in-app timer for a minimum duration before "complete" unlocks | Study sessions, focused trading study |
  | `integration` | Pulled automatically from a connected data source | Workout minutes/steps via Android Health Connect, screen-time via device usage APIs |
  | `location` | Must be within a saved GPS radius | Gym check-in, library/study spot |
  
  **Suggested mapping for your example quests:**
  
  - Workout → `integration` (Health Connect exercise minutes) or `photo` if you'd rather not connect health data
  - Morning prayer → `self_report` (some things don't need policing)
  - Study trading → `timer` (minimum focused minutes)
  - Post content → `link` (paste the post URL)
  - Read a book → `photo` (progress shot) or `timer`
  - Apply for jobs → `link` or `photo` (confirmation email/screenshot)
  
  **How it plugs into the schema:** `daily_quests.verification_type` + `verification_config` (jsonb) define the requirement when you create the quest (e.g. `{ min_minutes: 20 }` for timer, `{ lat, lng, radius_m }` for location). `quest_completions.proof_url` / `verified` / `verified_at` capture what was actually submitted. XP is awarded immediately for `self_report`, but for the stronger tiers the completion can be marked `verified: false` until the proof/integration check passes — so the XP toast still feels instant, while the "[System]" alert log can honestly show which completions were verified vs. self-reported if you ever want to see your own integrity stats.
  
  **Recommendation:** don't verify everything on day one — start most quests as `self_report` in Phase 1, then upgrade your highest-stakes quests (workout, study) to `photo`/`timer`/`integration` once the core loop is working. This is naturally a Phase 3–4 addition, not a blocker for MVP.

  ## 9. Stats System
  
  Each Daily/Main Quest optionally tags a `stat_target` + `stat_amount`. On completion, that stat's `points` increments. Stats have no cap and no separate "level" — they're a lifetime accumulation, visualized as a radar chart on the dashboard so you can see which parts of life are getting neglected at a glance.
  
  Optional future layer: derive a per-stat "level" the same way you derive character level, using a smaller XP curve, so stats feel more RPG-like individually (e.g. "Discipline Lv. 7").

  ## 11. AI Quest Master (Goal Planner) — Designed for Later Upgrade
  
  **Role clarity, since this changes how the whole app should feel:** the AI Quest Master isn't a side feature bolted onto an otherwise static app — it *is* the System. You're the **Architect** (you design the goals, own the app, ultimately decide what gets accepted) and the **Player** (you're the one out in the real world executing quests). The System is the third role — the admin — and once the AI Quest Master exists, it should be the single voice behind every "[System]" alert (§12), not just the goal-planning screen. That means:
  
  - Level-up, rank-up, and quest-issued alerts should eventually be phrased *by* the AI in-character as the System, not pulled from a static template
  - Missed-quest/punishment alerts get their bite from the same voice — consistent, a little cold, always addressing you as the Player
  - The Quest Master's goal-breakdown feature (below) is just the most functional expression of that same admin role — the System doesn't just talk strict, it actually plans and enforces
  
  Practically, this means Phase 1's static alert templates can ship first (so the core loop works without needing an LLM call on every event), and Phase 6 doesn't just add a planning screen — it upgrades the *voice* of every existing alert to route through the AI instead of a template, once the Quest Master exists.
  
  Fully feasible, and it slots in cleanly without touching the core schema — it just becomes another way rows get created in `main_quests`, `milestones`, and `daily_quests`.
  
  **How it works:**
  
  ```
  You type/speak a goal
    ("I want to pass a prop firm challenge")
          │
          ▼
  Edge Function calls an LLM (e.g. Claude via
  the Anthropic API) with the goal + your
  current stats/quest list as context, and a
  strict instruction to return ONLY structured
  JSON matching your schema:
    {
      main_quest: { title, description, xp_reward },
      milestones: [ { title, xp_reward }, ... ],
      suggested_daily_quests: [
        { title, xp_reward, difficulty, stat_target }, ...
      ]
    }
          │
          ▼
  App shows you the proposed plan as a review
  screen — edit/remove/accept items individually
          │
          ▼
  On accept, rows are inserted into main_quests /
  milestones / daily_quests exactly like if you'd
  created them by hand
  ```
  
  **Why this design works well for you specifically:**
  - The AI never writes directly to your DB — it proposes, you approve. Keeps you in control and avoids junk quests polluting your list.
  - The API call happens server-side (Supabase Edge Function), so your API key never sits in the mobile app bundle.
  - Because it outputs the *same shape* your manual quest-creation forms already produce, this is an additive feature — nothing in Phases 0–5 needs to change to support it later. You can genuinely ship the manual version first and bolt this on afterward.
  - Optional nice extension once this exists: a periodic "Quest Master check-in" push notification — the AI reviews your recent completions/stats and proposes a new quest, rather than you always having to initiate it.
  
  **What it needs when you build it:** an LLM API key (Anthropic API works well for this — strong at structured JSON output and reasoning about multi-step plans), one Edge Function endpoint, and a review/accept UI screen. No schema changes required.

  ## 12. "System" Alert / Notification Design
  
  This is the signature feel of the app — the mechanic that makes it feel alive instead of like a checklist.
  
  **Two layers:**
  
  1. **OS push notifications** (via Expo Notifications + Supabase Edge Function cron): fire even when the app is closed. Examples:
     - Morning: *"[System] Daily Quests have been issued."*
     - Evening, quest still incomplete: *"[System] Warning: 'Workout' quest incomplete. Time remaining: 3h."*
     - Streak at risk: *"[System] Your streak will end in 2 hours."*
  2. **In-app full-screen "System" overlay** (rendered from `components/system-alert/`): triggered when the app is opened after a push, or immediately after a significant in-app event. Mimics the Solo Leveling blue-panel "System" window — dark translucent overlay, gold/white monospace-ish text, typewriter reveal animation, single "Accept" tap to dismiss. Used for:
     - New daily quests issued (start of day)
     - Level up ("[System] You have reached Level 12.")
     - Rank up ("[System] Rank promoted: Scout → Warrior.")
     - Main Quest milestone or full completion
     - Missed quest / streak broken (a harsher, red-tinted variant — this is the "consequence" beat that makes it feel real)
  
  **Why the split:** push notifications are what make it feel like a system is watching your life even when the phone's locked; the full-screen overlay is what makes the moment of opening the app feel like a cutscene, not a status check.
  
  **Tech notes:** Expo Notifications handles device push tokens and OS-level delivery; a Supabase Edge Function on a cron schedule checks each user's incomplete quests/streak status daily and calls the Expo Push API. In-app overlays are just local state driven by a queue — any XP-engine event (level up, quest complete, streak broken) pushes an alert object onto the queue, and the overlay component renders the next one in line.

  ## 13. Smart Glasses Integration (Future) — Feasibility
  
  Realistic, and it splits into two tiers depending on how deep you want to go — worth knowing both now so the notification system (§11) is built in a way that doesn't block either path later.
  
  **Tier 1 — Notification mirroring (easy, works with most Android-compatible glasses today):**
  Most current Android smart glasses (e.g. Even Realities G1, Meta Ray-Ban Display, various Android XR-based glasses) work by mirroring your phone's notifications to a small HUD via Bluetooth, using a companion app. Since your app already fires real Android push/local notifications for "[System]" alerts, quest-issued messages, and level-ups, those notifications would **already show up on this class of glasses automatically** — no extra integration work needed, as long as the glasses' companion app is configured to mirror app notifications. This tier is essentially "free" once Phase 3 (notifications) is done.
  
  **Tier 2 — Native glasses app / custom HUD (harder, hardware-dependent):**
  Some platforms (notably Android XR, Google's glasses/headset platform announced for 2025–2026 devices, and vendor-specific SDKs like Even Realities' own SDK) let you build a dedicated glasses-side app or display card — e.g. a persistent mini-HUD showing your XP bar, current quest, or streak directly on the lens, not just a notification popup. This requires:
  - Picking a specific glasses platform/SDK once you own the hardware (SDKs vary a lot and aren't interchangeable)
  - A lightweight "glanceable" UI package separate from the phone app, reading the same Supabase backend
  - No changes to the core schema or XP engine — this tier is purely a new display client on top of what you're already building
  
  **Recommendation:** don't design for a specific glasses SDK now — hardware and SDKs in this space are still shifting fast. Just make sure (as already planned) that meaningful events flow through real device notifications, and treat glasses as a future "second client" reading the same backend, the same way the AI Quest Master is a future add-on.

  ## 14. Avatar System
  
  **Onboarding:** you enter height/weight/body metrics manually (a photo alone can't measure these accurately — no reference scale to calibrate against). You also upload a full-body photo purely for art generation, and set your calisthenics goal.
  
  **Avatar generation:** an Edge Function sends your photo to an image-generation API (img2img/style-transfer) with a fantasy-system art style prompt, producing one stylized base avatar. This becomes your permanent character art — not regenerated from scratch later, since repeated AI regeneration drifts and won't look like "the same character" level to level.
  
  **Progression:** instead of regenerating the whole avatar, progress is shown through **layered overlays** on the same base art — armor pieces, an aura/glow effect, and shading changes — unlocked at level/stat thresholds (e.g. Discipline/Health stat milestones tied to your calisthenics goal). This is cheap, fast, and visually consistent, and mirrors exactly how manhwa "system" characters visibly level up without becoming a different person each panel.
  
  **Schema:**
  ```sql
  avatars (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    source_photo_url  text,      -- original upload, private storage
    base_avatar_url   text,      -- generated stylized art
    height_cm     numeric,
    weight_kg     numeric,
    goal_type     text,          -- e.g. 'calisthenics'
    current_tier  int default 1, -- drives which overlays render
    updated_at    timestamptz default now()
  )
  
  avatar_overlays (
    id            uuid primary key default gen_random_uuid(),
    tier          int not null,           -- unlock threshold
    overlay_type  text,                   -- 'armor' | 'aura' | 'shading'
    asset_url     text
  )
  ```

  ## 15. G-Coin Economy & Shop
  
  A second currency alongside XP — XP drives level/rank (long-term progression), G-coin is spendable pocket money for the shop (short-term reward loop). Earned as a side-effect of quest completion (e.g. `gcoin_reward` alongside `xp_reward` on each quest).
  
  **Schema:**
  ```sql
  -- add to profiles: gcoin_balance int default 0
  
  gcoin_log (               -- immutable ledger, same pattern as xp_log
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    amount        int not null,           -- positive (earn) or negative (spend)
    source_type   text,
    source_id     uuid,
    created_at    timestamptz default now()
  )
  
  shop_items (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),   -- you define your own shop
    title         text not null,
    description   text,
    cost_gcoin    int not null,
    item_type     text check (item_type in ('cosmetic','real_world_reward','consumable')),
    active        boolean default true
  )
  
  purchases (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    item_id       uuid references shop_items(id),
    cost_paid     int not null,
    purchased_at  timestamptz default now()
  )
  ```
  
  Since it's your own personal system, you define the shop items yourself — cosmetic ones (avatar overlays, HUD themes/titles) and real-world ones (e.g. "300 G-coin = guilt-free takeout night," "1000 G-coin = new game"). The AI Quest Master (§11) could later auto-suggest reward items scaled to a goal's difficulty, but that's an easy bolt-on, not needed for MVP.

  ## 16. "Mana Liquid" Consumable Log
  
  A flavor layer over a simple food/water/supplement log — no new mechanic, just reskinned copy and an optional generated description.
  
  ```sql
  consumables_log (
    id            uuid primary key default gen_random_uuid(),
    user_id       uuid references profiles(id),
    logged_at     timestamptz default now(),
    category      text check (category in ('meal','water','supplement')),
    item_name     text,           -- what you actually ate/drank/took
    flavor_text   text,           -- generated "mana potion" description
    stat_target   text,           -- optional: ties nutrition to a stat, e.g. Health
    stat_amount   int default 0
  )
  ```
  
  Flavor text can be generated the same way as the AI Quest Master — a small Edge Function call that turns "chicken breast and rice" into something like *"[System] Mana Potion consumed: +Strength Essence. Vitality restored."* Purely cosmetic, no gameplay logic depends on the wording.

  ## 17. Workout & Supplement Guidance
  
  A content library, not a diagnostic engine — general, widely-known fitness/nutrition information (standard calisthenics progressions, common protein-per-bodyweight guidance, typical creatine dosing) surfaced contextually based on your calisthenics goal and current stats. This is educational content, not a personalized medical or nutrition plan — worth treating any supplement guidance as "common practice" info and checking with a doctor if you have any health conditions or take medication, same as you'd want for any fitness app.
  
  Practically: a `workout_programs` / `guidance_content` table you (or the AI Quest Master later) populate, surfaced on the dashboard or as suggested Daily Quests tied to your goal.

  ## 18. Punishment / Discipline System
  
  Two real implementation paths on Android — worth being upfront about the tradeoff:
  
  | Approach | What it can actually do | Setup cost | Risk |
  |---|---|---|---|
  | **Accessibility-Service app blocker** (recommended) | Hard-blocks specific apps (social media, games, whatever you name) until a quest is completed. Emergency calling/essentials always stay accessible. | Just a permission grant, no special provisioning | Low — worst case you disable the permission |
  | **Device Owner mode** (full lockout) | Full-device lock, like enterprise MDM/parental controls | Must be provisioned via ADB *before* first Google sign-in on the device | High — misconfigure it and you may need a full factory reset to regain access |
  
  **Recommendation:** build the Accessibility-Service blocker. It's genuinely strict (you cannot open the blocked apps, no workaround from inside them) without carrying the risk of locking yourself out of your own phone over a missed habit.
  
  **Punishment tiers (config, not schema — logic lives in the Edge Function that runs the daily rollover):**
  - Missed 1 daily quest → blocked apps (your choice list) until first quest of the next day is completed
  - Broken streak → longer block window + G-coin penalty + a stat "decay" tick
  - Missed Main Quest milestone deadline (if you set one) → System sends a harsher red-tinted alert, no XP penalty by default (main quests are long-term, punishing missed deadlines too hard tends to just make people quit instead of restart)
  
  ```sql
  punishment_rules (
    id             uuid primary key default gen_random_uuid(),
    user_id        uuid references profiles(id),
    trigger_type   text check (trigger_type in ('missed_quest','broken_streak','missed_milestone')),
    blocked_apps   text[],       -- package names to block
    block_duration_hours int,
    gcoin_penalty  int default 0,
    active         boolean default true
  )
  ```

  ## 19. Trait System
  
  Permanent passive achievements earned through *consistent* behavior over time — distinct from single quest completions, and the mechanism that rewards showing up repeatedly rather than any one good day.
  
  Traits are defined once (`traits` table, §2) with an `unlock_condition` (e.g. "100 quests tagged workout completed"), a rarity tier (common → legendary), and an optional `passive_bonus` (a small permanent stat or XP effect — e.g. "Iron Body" could grant +5 Health permanently). A daily/nightly Edge Function job checks unlock conditions against your completion history and inserts into `user_traits` when met, firing a "[System] Trait Unlocked" alert.
  
  Example progression:
  - 100 Workouts → **Iron Body** (uncommon)
  - 100 Trading Sessions → **Market Student** (uncommon)
  - 30 Consecutive Days Posting Content → **Consistent Creator** (rare)
  - 365 Days Active → **Unbroken Will** (legendary)
  
  Traits are visible on the Traits & Titles screen (§10) — this is the "long-term identity" layer that Level/Rank alone doesn't capture, since a level resets in feel every time you level up again, but a Trait, once earned, is permanent proof of who you've become.
  
  ---
  
  ## 20. Titles System
  
  Titles are separate from Rank: **Rank measures how far you've progressed overall; Titles represent specific things you've accomplished.** You can only display one at a time, so equipping one is itself a small identity statement.
  
  Defined in `titles`, unlocked into `user_titles` the same way traits are — either standalone (via an `unlock_condition`) or as a reward attached to a Hidden Achievement (§21). `profiles.active_title_id` holds the currently equipped one, shown right under Rank on the Status Window:
  
  ```
  Rank: Elite
  Title: Empire Architect
  ```
  
  Example titles: **The Disciplined**, **The Builder**, **The Scholar**, **The Trader**, **The Survivor**, **Empire Architect** — you can define your own set matching your actual goals (trading, relocation, building this app itself) rather than generic RPG flavor.
  
  ---
  
  ## 21. Hidden Achievements System
  
  Achievements with `hidden: true` (§2 schema) simply aren't rendered anywhere in the UI — not in a locked/silhouette state, not listed at all — until their `unlock_condition` is met. This is what makes unlocking one feel like a genuine surprise rather than a checklist you were quietly grinding toward.
  
  Checked by the same nightly Edge Function job as Traits, against whatever data the condition needs (streaks, cumulative quest counts, specific stat thresholds, even something like "logged before 6 AM for 30 days" pulled from `quest_completions.completed_on` timestamps). On unlock: a distinct, more dramatic "[System]" alert variant (longer reveal animation, unique sound) plus XP/G-coin/title rewards as configured on the achievement row.
  
  Example hidden achievements: Wake before 6AM for 30 days, complete 100 total quests, earn your first $100 online, run 100km total, reach Level 50 — a mix of behavioral consistency and milestone thresholds.

  ## 22. Progression Philosophy
  
  Worth stating explicitly since it should guide every feature decision going forward: **this app is not a checklist — it's identity transformation.** You're not tracking tasks, you're becoming a different version of yourself through repetition, and every system above exists to make that legible:
  
  ```
  Action → Quest Completion → XP Gain → Level Increase
         → Stat Growth → Traits Unlocked → Titles Earned
         → Identity Evolution
  ```
  
  When evaluating any future feature (including whatever comes after Phase 7), the test is: does this reinforce that loop, or is it just another number on a screen? XP/Level answer "how much have I done." Stats answer "in what direction." Traits and Titles answer "who have I become." All three matter; none of them replace the others.
