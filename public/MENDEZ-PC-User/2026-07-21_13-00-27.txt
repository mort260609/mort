  # Implement Multisig + Timelock Admin Structure
  
  Per the locked design (NEXAI_Multisig_Timelock_Design.pdf): move admin
  control from a single EOA to a 2-of-3 Gnosis Safe with a 48-hour
  timelock on governance changes, plus a 1-of-3 guardian pause mechanism.
  Signers/guardians are provisional placeholders for now.
  
  ## Part 1: Multisig
  
  Use Gnosis Safe (Safe{Wallet}) — do not build a custom multisig
  contract. Deploy via Safe's standard factory
  (`SafeProxyFactory`/`Safe` singleton on Base Sepolia — look up current
  addresses, don't assume).
  
  - Threshold: 2-of-3
  - Signers: 3 placeholder addresses (can be freshly generated test
    addresses for now — clearly commented as provisional, to be
    replaced with real signers later)
  
  ## Part 2: Timelock
  
  Use OpenZeppelin's `TimelockController` — do not write custom timelock
  logic. This is exactly the audited-primitive-over-custom-code pattern
  used throughout this project.
  
  - Delay: 48 hours
  - Proposer role: the Safe (2-of-3 multisig)
  - Executor role: permissionless (per the design doc's recommendation
    — anyone can execute once the delay passes, but only what was
    proposed and approved)
  - Applies to: `updateBasket`, `setTokenConfig`, any future role grants
    on `NexaiCore`
  
  **Migration**: `NexaiCore`'s `DEFAULT_ADMIN_ROLE` and `ADMIN_ROLE`
  need to be transferred from the current EOA to the `TimelockController`
  address. Write this as a clearly separate script
  (`script/MigrateAdminToTimelock.s.sol`), not bundled into existing
  deploy scripts — this is a significant, one-way action worth its own
  explicit step.
  
  ## Part 3: Guardian Pause
  
  - Add `Pausable` (OpenZeppelin) to `NexaiCore` — `deposit()` and
    `redeem()` should revert when paused
  - Guardian role: 1-of-3 of the same Safe signers can trigger pause
    (fast, no delay)
  - **Unpausing goes through the timelock** (2-of-3 + 48h) — do not let
    pause and unpause share the same fast path
  - Implement pause/unpause as functions gated appropriately: pause
    callable by any Safe signer directly (or via a lower-threshold
    mechanism — flag if Safe's standard tooling doesn't cleanly support
    a 1-of-3 sub-threshold and a design adjustment is needed), unpause
    only callable by the TimelockController
  
  ## Part 4: Cancel Mechanism (per Dmitry's review feedback)
  
  Per the design doc update: pause and cancelling a bad queued proposal
  are separate tools. Confirm `TimelockController`'s built-in `cancel()`
  function (exists in OZ's implementation) is accessible to the same
  1-of-3 guardian threshold, separate from the pause mechanism, acting
  on the proposal rather than on Core's deposit/redeem functions.
  
  ## Tests
  
  - Confirm the EOA can no longer directly call admin functions on
    `NexaiCore` after migration
  - Confirm a proposal through the Safe → Timelock → execute flow works
    end-to-end (propose, 2-of-3 approve, wait/warp past 48h, execute)
  - Confirm 1-of-3 can pause instantly
  - Confirm unpause requires the full timelock flow, not 1-of-3
  - Confirm a queued proposal can be cancelled by 1-of-3 without
    requiring pause
  - Confirm a queued proposal is NOT automatically halted by pause
    (per the design doc: these are deliberately separate)
  
  ## Notes
  
  - Testnet only, no mainnet
  - This is genuinely new infrastructure, not a modification of
    `NexaiCore`'s existing mint/redeem logic — treat with the same
    rigor as the original core contracts, not as a quick add-on
  - Signers/guardian addresses are provisional — comment clearly
    wherever they appear
