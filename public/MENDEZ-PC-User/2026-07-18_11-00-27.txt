  # Harden Test Coverage — Realistic Edge Cases (Option 1)
  
  Per Dmitry: solid coverage of realistic edge cases we might actually
  hit during normal use, not a full auditor-style adversarial pass (that
  comes later, once multisig/timelock and M2 basket scaling are settled
  — see the design doc already sent).
  
  ## Scope
  
  Add tests across the existing suite covering:
  
  ### 1. Small / dust amounts
  - Minting NEXAI amounts small enough that `requiredWeth` or
    `requiredUsdc` could round to a very small number (but still > 0)
    — confirm the round-up formula still behaves correctly at the low
    end, not just at "normal" amounts like 1-10 NEXAI
  - Confirm the zapper's existing `require(requiredAmount > 0, ...)`-style
    guards (already in `NexaiCore.deposit`) are actually exercised by a
    dedicated test, not just implied by other passing tests
  
  ### 2. Repeated/sequential operations
  - Multiple deposits and redeems in sequence from the same user,
    confirming balances track correctly across many operations (not
    just the single deposit-then-redeem pattern most existing tests use)
  - Interleaved deposits from *different* users (not just Alice) to
    confirm the system correctly handles multiple depositors without
    cross-contamination of balances or accounting
  
  ### 3. Basket reconfiguration during normal operation
  - Test that a `deposit()`/`zapDeposit*()` call correctly uses whatever
    basket configuration is live *at the time of the call* — not a
    cached/stale one — by changing the basket via `updateBasket()`
    between two deposits and confirming the second deposit uses the new
    weights
  - Test that redeeming after a basket change correctly returns the
    *current* basket assets (per the earlier discussion with Dmitry
    about what "redeem" actually returns) — this is a good one to add
    since it directly verifies a behavior we've explained to him but
    never explicitly tested
  
  ### 4. Zapper-specific realistic cases
  - A zap deposit immediately followed by a zap deposit of a *different*
    entry token (e.g. USDC then WETH then USDT in sequence) — confirms
    no state leaks between the three entry paths
  - Slippage buffer edge case: a deposit where the actual swap cost
    lands very close to (but under) `maxUsdcIn`/`maxWethIn`/`maxUsdtIn`
    — confirms the refund math is exactly right at the boundary, not
    just in the comfortable-margin cases existing tests use
  
  ## Out of Scope (Deferred to the Later Deep Pass)
  - Extreme/adversarial fuzz ranges beyond what's already in
    `testFuzz_Solvency`
  - Access-control stress testing beyond what's already covered
  - Multi-step attack scenario simulation
  - Anything assuming the multisig/timelock or M2 basket structure,
    since those aren't built yet
  
  ## Notes
  - Add these to the existing test files (`NexaiCore.t.sol`,
    `NexaiZapper.t.sol`) rather than creating new files, unless a
    specific area genuinely warrants its own file
  - Standard process: run `forge build && forge test` yourself before
    reporting back — same as every round so far

  Two things worth your attention beyond "tests added":
  
  A real gap I found while writing the reconfiguration tests, not just a test: test_Redeem_RevertsWhenBasketReconfiguredToNewUnfundedToken in NexaiCore.t.sol demonstrates that if updateBasket() ever introduces a token Core doesn't already hold (as opposed to just changing weights on the same two tokens), every existing depositor's redeem() call reverts — because updateBasket only rewrites the weight/token bookkeeping, it never actually moves tokens into Core.
