  ## KNOWN OPEN ISSUES — please prioritize these
  
  ### 1. Typing indicator — has been rewritten multiple times, still has bugs
  This feature has a long, painful history across many sessions. The architecture SHOULD be:
  - A self-contained hook/local-ref based approach (NOT module-level singleton state — a module-level singleton caused a "zombie channel" bug where the indicator would work once then never again)
  - Sender: on keystroke, if not already marked typing, broadcast isTyping=true via Supabase Presence track(); reset a 3-second timer on every keystroke; when timer fires (or on send/unmount), broadcast isTyping=false
  - Receiver: simply mirrors whatever Presence sync state says, no separate timeout logic on the receiving end
  - Displayed in TWO places: (1) a pulsing dot bubble inside the chat (ListFooterComponent), and (2) header text ("Active now" / "Last seen [time]" / "Typing...") — both should exist, this was an explicit decision to keep both, not replace one with the other
  
  **Current known bug**: the indicator gets stuck showing "typing" even after the other person has genuinely stopped typing — the auto-clear timer isn't reliably firing or its broadcast isn't reaching the other side. Please investigate fresh, and importantly, check whether your own prior changes to this feature (before this handoff) may have altered the architecture described above in a way that reintroduced this bug.
  
  ### 2. Message not displaying after restarting a previously-deleted chat
  Scenario: user deletes a chat conversation (soft delete via chat_members.deleted_at), then searches for that same contact and starts what appears to be a "new" conversation, sends a message — the message doesn't display in the chat.
  
  Likely cause: the chat-creation/find-or-create logic probably finds the existing (soft-deleted) chat row and reuses it, but something in the message-fetch or chat-open flow may still be treating it as "deleted" and filtering out content. Likely fix: when reopening/messaging a previously-deleted chat, clear deleted_at for the current user's chat_members row as part of that flow (similar to how archive auto-clears on new messages).
