  # Nexora System — The Life OS
  ### Product Specification (v1)
  
  ## 1. Concept
  
  Nexora System turns real-life goals into an RPG — generating adaptive daily quests, tracking stats that grow from real progress, and rewarding proof of actual skill growth. Inspired by "System" narratives (e.g. Solo Leveling), the app should feel like a living System speaking to the player, not a SaaS dashboard.
  
  Built solo (founder + Claude for planning, Claude Code for execution, Stitch/Figma MCP for UI where needed), bootstrapped, full product (not a stripped-down MVP).
  
  ---
  
  ## 2. Tech Stack
  
  **Frontend:** React Native (Expo), React Native Reanimated + React Native Skia for custom animations/visual effects.
  
  **Backend:** Supabase (Postgres — DB, Auth, Storage), Node.js/TypeScript API layer.
  
  **AI Admin layer:** Claude API — handles quest generation, verification review, and System dialogue. Kept as an isolated service so prompts can iterate independently of core data logic.
  
  Rationale: single language (TypeScript) across the stack for AI-assisted solo development; Postgres for transactional integrity in the points/economy system; React Native chosen over Flutter for better Claude Code support and Figma/Stitch MCP compatibility.
  
  ---
  
  ## 3. Onboarding Flow (6 screens)
  
  1. **Initialize System** — boot-up/tone-setting visual.
  2. **Username & Password** — styled auth input.
  3. **Loading / Registration Sequence** — typewriter text: "Registering user...", "Logging in user...", "User logged in." → fades to "Welcome to the Nexora System, [username]" → fades to Status Window teaser with `??` placeholders on Name, Title, Level, Trait, Stats.
  4. **Data Input Popup** (~2s after teaser) — collects: real name, Main Quest (yearly goal), body weight/baseline stats, constraints (time available, equipment/resources). Side/Daily Quests are NOT manually entered — they auto-generate from the Main Quest.
  5. **Syncing** — "Syncing user to the System..." processing moment.
  6. **Status Window (Populated)** — same screen as step 3's teaser, now showing real Name, Level 1, and initial stats. This is the payoff beat.
  
  ---
  
  ## 4. Status Window (Home Base)
  
  Single-anchor design: the Status Window is the only full screen. Everything else (Shop, Quest details, etc.) is a modal layered on top of it.
  
  **Layout:**
  - Header: Name, Title, Level + EXP bar
  - Rank letter (E–S scale) — top right, visible always
  - Core Stats (STR/DEX/INT/VIT/etc., mapped to real-world goal categories)
  - Attributes (behavior-derived, see §8)
  - Traits (milestone-unlocked; locked ones hidden until first unlock)
  - Quest lines (compact): `Main Quest: —`, `Side Quest: —`, `Daily Quest: —`, each with an animated pulsing/dotting indicator. Tap → modal with full quest details (description, requirements, reward breakdown).
  
  ---
  
  ## 5. Quest System
  
  **Hierarchy:** Main Quest (yearly) → Side Quests (auto-generated checkpoints, System-timed) → Daily Quests (adaptive, generated daily).
  
  **Daily Quest:**
  - AI-generated based on onboarding data + rolling inputs (streak, yesterday's completion, 7-day completion rate, time available, current Side Quest progress).
  - Difficulty target: ~80% success probability, adapting up/down based on recent performance.
  - Reward: ~15–30 points + EXP.
  - **Completed** → celebratory popup → disappears → hard reset, new one at 00:00.
  - **Uncompleted** → does NOT just vanish at midnight — routes through the Streak Freeze/Grace mechanic first (see §6) before resolving.
  
  **Side Quest:**
  - Auto-generated from the active Main Quest; regenerates when the System determines it's time (not a fixed daily reset).
  - Has its own deadline (time-limited, not open-ended).
  - Reward: ~100–200 points + larger EXP chunk.
  - **Deadline outcome:** under 50% progress = failure window shown, quest clears, new one generates. 50%+ progress = grace extension granted (extra time), completes normally with celebration when finished.
  
  **Main Quest:**
  - The yearly goal set at onboarding. Persists until genuinely completed or majorly revised.
  - Reward: 500–1,000+ points + big EXP + guaranteed Skill Card or Trait unlock.
  
  ---
  
  ## 6. Streak & Grace Mechanic
  
  - Streak Freeze/Shield — earn periodically (or via points), auto-protects a missed day.
  - Grace period — a make-up window still counts, reduced EXP.
  - Soft-fail vs hard-fail — partial effort still earns some EXP without breaking streak; only total no-shows risk the streak.
  - Distinct visual cue when a freeze/grace triggers (different from both Complete and Fail) — player must clearly understand "you were saved."
  
  ---
  
  ## 7. Title & Rank (separate systems)
  
  **Title** — fantasy-style, unisex (one title per tier, no gendered variants), driven by **Level** (frequent, granular progression). Names should feel grounded/relatable, not Gen-Z slang:
  
  | Level Range | Title |
  |---|---|
  | 1–10 | Novice |
  | 11–30 | Laborer |
  | 31–60 | Craftsman |
  | 61–100 | Warrior |
  | 101–150 | Knight |
  | 151–200 | Vanguard |
  | 201+ | Sovereign |
  
  **Rank** — E through S letter scale, a separate and rarer marker of overall standing (not tied 1:1 to Level/Title).
  
  ---
  
  ## 8. Attributes vs Traits
  
  - **Attributes** — current/ongoing behavioral state. AI-generated from real patterns. Can strengthen OR fade based on recent behavior (e.g. "Disciplined" tied to an active streak — can drop off if consistency slips). Unlocked states, not numeric bars.
  - **Traits** — permanent milestone badges. AI-generated from milestone events. Once earned, never lost — a historical record of achievement. Locked traits are hidden (not shown as silhouettes) until first unlock.
  
  ---
  
  ## 9. Shop & Auction House
  
  Launched as a popup modal from the Status Window (X icon to close), background dims/blurs. Two tabs within one modal: **Normal Shop** and **Auction**.
  
  **Skill Cards:**
  - Flippable (tap/hold), browsable before purchase — owned or not.
  - Front: art/icon, name, rarity tier, lock status.
  - Back: EXP boost granted, real-world benefit of the skill, proof requirement, point cost.
  - Flip happens BEFORE purchase (full info available while browsing).
  
  **Flow: Buy-then-prove.** Purchase unlocks ownership; the EXP boost stays inactive until proof of the actual skill is submitted and verified. Button state: "Buy" (not owned) → "Locked — Submit Proof" (owned, unproven) → active/equipped (proven).
  
  **Rarity tiers, cost, and level gates:**
  
  | Rarity | Point Cost | Level Gate |
  |---|---|---|
  | Common | 100 | L1+ |
  | Uncommon | 300 | L10+ |
  | Rare | 750 | L30+ |
  | Epic | 1,500 | L60+ |
  | Legendary | 3,000 | L100+ |
  | **Mythical** | 6,000+ | **L200+, NOT purchasable in Normal Shop — auction or earned-reward only** |
  
  All cards are visible in the shop regardless of player level; shown locked/unpurchasable if the level requirement isn't met.
  
  **Mythical special rule:** bids from players under Level 200 do not go through — they revert automatically at the bid level.
  
  **Auction House:**
  - Seller-chosen listing duration.
  - Both proven and unproven cards can be listed.
  - System takes a **5% cut, sale-based** (taken from final sale price, NOT an upfront listing fee) — framed in-fiction as "the Architect's" cut.
  - Future scope (not v1): trading other in-game items (e.g. rare weapons) via the same auction mechanism.
  
  ---
  
  ## 10. Points Economy — NXR
  
  **NXR is closed-loop for launch:**
  - Earned only (quests, streaks, milestones) — cannot be purchased with real money or crypto at launch.
  - Freely transferable peer-to-peer between users.
  - Spendable in Shop/Auction.
  - **No cash-out to real money or crypto at launch.** This is explicitly flagged as a future phase requiring proper legal/regulatory setup (money transmitter licensing considerations; Nigeria's July 2026 Virtual Assets Coordination executive order significantly tightened crypto/VASP licensing requirements). Do not build a cash-out feature without dedicated legal review first.
  
  **Earn rates:**
  - Daily Quest: ~15–30 points
  - Side Quest: ~100–200 points
  - Main Quest/milestone: 500–1,000+ points + guaranteed card/trait unlock
  
  ---
  
  ## 11. Verification & the AI Admin
  
  The AI Admin is "the System" itself — the in-fiction voice the player interacts with. It has four jobs:
  
  1. **Quest Generation** — daily/side/main quests, adaptive to stats/streak/performance.
  2. **Verification Review** — reviews submitted proof, returns pass/fail. **AI judgment is final — no manual review/appeal fallback.**
  3. **Dialogue/Flavor Text** — consistent System voice for quest text, completion messages, level-ups (stern but fair tone).
  4. **Anti-abuse logic** — flags suspicious patterns (reused proof, impossible progress jumps).
  
  **Verification hierarchy (strongest to weakest):**
  1. **Integration data** (GitHub commits, wearables, language-app APIs) — hardest to fake, no AI judgment call needed. Even when auto-verified this way, the AI Admin is still the one that confirms and announces completion via the popup — the player experience is always mediated through the System's voice, regardless of underlying verification method. (Post-launch phase feature — real engineering lift per integration.)
  2. **AI photo/video review** — for goals without a clean integration path (e.g. calisthenics form-check). AI judgment final.
  3. **Honor system** — Daily Quests only; low stakes, self-report.
  
  ---
  
  ## 12. Visual Style Guide
  
  **Palette:** Deep charcoal/near-black base (not pure black), gold/amber glow accents. Gold ties thematically into achievement, rank, and title progression.
  
  **Typography:**
  - Display/stats: **Rajdhani**
  - Body/descriptions: **Inter**
  
  **Visual language:**
  - Frosted glass panels with thin glowing borders
  - Subtle scan-line/particle shimmer on panel edges
  - Background dims/blurs when any modal opens
  - Sharp geometric corners (not soft/rounded)
  - Numbers tick/count up rather than snapping instantly
  
  **Motion:**
  - Modals: fade + slight scale-in (not slide-up)
  - System messages: typewriter text
  - Level-Up: distinct, high-polish animation — flash/glow pulse, ticking numbers, possible shockwave/ring effect (biggest dopamine beat in the app)
  - Quest Complete: popup "stamps" in reward numbers count up
  - Quest dot: continuous subtle pulsing/breathing animation
  - Card Flip: physical 3D-style flip, not a crossfade
  - Streak Freeze/Grace: distinct visual cue, different from both Complete and Fail
  - Rank Up/Title Change: rarer and bigger than a normal level-up — near-fullscreen takeover moment
  
  **Icons:** Custom icon set (not generic UI icons) — emblem/rune-styled icons for stats, quest markers as waypoints/objectives, distinct NXR currency icon. Consistent line-weight + gold accent throughout. Locked traits/cards use silhouette/outline versions.
  
  *(Detailed icon/animation specs to be finalized once core screens are built — treat this section as style direction, not final asset list.)*
  
  ---
  
  ## 13. Build Phases
  
  **Phase 1 — Foundation & Core Loop**
  - Expo (React Native) project setup, Supabase project (DB/Auth/Storage) connected via `.env`
  - Row Level Security (RLS) policies enabled before real user data is written
  - Initial schema: `users`, `stats`, `quests`
  - Onboarding flow (all 6 screens, static UI first)
  - Basic auth (username/password)
  - Status Window (static first — no live quest generation yet)
  
  **Phase 2 — Quest System**
  - Daily Quest AI generation + completion + honor-system verification
  - EXP/points awarding, level-up logic + animation
  - Streak tracking + freeze/grace mechanic
  - Side Quest generation (System-timed) + deadline/grace logic
  - Main Quest tracking
  
  **Phase 3 — Identity Layers**
  - Title (Level-driven) + Rank (E–S) display and progression
  - Attributes + Traits (AI-generated)
  
  **Phase 4 — Economy**
  - NXR points system (earning, balances, P2P transfer)
  - Normal Shop (Skill Cards, buy-then-prove, flip interaction, level gates)
  - AI photo/video proof verification for Skill Cards
  
  **Phase 5 — Auction House**
  - Listings, bidding, seller-chosen duration, 5% sale fee
  - Mythical tier rules (Level 200+ gate, bid revert logic)
  
  **Phase 6 — Polish & Depth**
  - Full icon set + animation polish
  - Integration-based verification (GitHub, wearables, etc.)
  
  ---
  
  ## 14. Explicitly Out of Scope for v1
  
  - Real-money or crypto cash-out for NXR (requires legal/regulatory review first)
  - Trading of non-card in-game items (future scope)
  - Manual/appeal review of AI verification decisions
  - Integration-based verification (Phase 6, not launch)

  -- Nexora System — Phase 1 Database Schema
  -- Tables: users, user_stats, quests
  -- Run this in the Supabase SQL editor after project creation.
  
  -- ============================================================
  -- USERS
  -- Extends Supabase's built-in auth.users with Nexora-specific profile data.
  -- ============================================================
  create table public.users (
    id uuid primary key references auth.users(id) on delete cascade,
    username text unique not null,
    name text not null,
    level int not null default 1,
    exp int not null default 0,
    exp_to_next_level int not null default 100,
    rank text not null default 'E' check (rank in ('E', 'D', 'C', 'B', 'A', 'S')),
    body_weight numeric,
    time_available_minutes int,           -- daily time constraint from onboarding
    main_quest_id uuid,                   -- set after main quest is created (fk added below)
    streak_count int not null default 0,
    streak_freezes_available int not null default 0,
    points_balance int not null default 0, -- NXR balance
    created_at timestamptz not null default now(),
    updated_at timestamptz not null default now()
  );
  
  -- ============================================================
  -- USER STATS
  -- Core RPG stats (STR, DEX, INT, VIT, WIS, etc.) — normalized so new
  -- stat types can be added without a schema migration.
  -- ============================================================
  create table public.user_stats (
    id uuid primary key default gen_random_uuid(),
    user_id uuid not null references public.users(id) on delete cascade,
    stat_key text not null,               -- e.g. 'STR', 'DEX', 'INT', 'VIT'
    value int not null default 0,
    updated_at timestamptz not null default now(),
    unique (user_id, stat_key)
  );
  
  -- ============================================================
  -- QUESTS
  -- Covers Main, Side, and Daily quests via the `tier` column.
  -- Only one active Main/Side/Daily quest per user is expected at a time —
  -- enforce that in application logic (or add a partial unique index later).
  -- ============================================================
  create table public.quests (
    id uuid primary key default gen_random_uuid(),
    user_id uuid not null references public.users(id) on delete cascade,
    tier text not null check (tier in ('main', 'side', 'daily')),
    title text not null,
    description text not null,
    status text not null default 'active'
      check (status in ('active', 'completed', 'failed', 'grace_period')),
    progress numeric not null default 0,   -- 0-100, used for Side Quest 50% grace rule
    reward_points int not null default 0,
    reward_exp int not null default 0,
    verification_method text not null default 'honor'
      check (verification_method in ('honor', 'ai_review', 'integration')),
    deadline timestamptz,                  -- null for Main Quest (open-ended until done)
    created_at timestamptz not null default now(),
    completed_at timestamptz
  );
  
  alter table public.users
    add constraint fk_main_quest foreign key (main_quest_id) references public.quests(id);
  
  -- ============================================================
  -- INDEXES
  -- ============================================================
  create index idx_quests_user_tier_status on public.quests (user_id, tier, status);
  create index idx_user_stats_user on public.user_stats (user_id);
  
  -- ============================================================
  -- ROW LEVEL SECURITY
  -- Users can only read/write their own data.
  -- ============================================================
  alter table public.users enable row level security;
  alter table public.user_stats enable row level security;
  alter table public.quests enable row level security;
  
  create policy "Users can view own profile" on public.users
    for select using (auth.uid() = id);
  create policy "Users can update own profile" on public.users
    for update using (auth.uid() = id);
  
  create policy "Users can view own stats" on public.user_stats
    for select using (auth.uid() = user_id);
  create policy "Users can update own stats" on public.user_stats
    for update using (auth.uid() = user_id);
  create policy "Users can insert own stats" on public.user_stats
    for insert with check (auth.uid() = user_id);
  
  create policy "Users can view own quests" on public.quests
    for select using (auth.uid() = user_id);
  create policy "Users can update own quests" on public.quests
    for update using (auth.uid() = user_id);
  create policy "Users can insert own quests" on public.quests
    for insert with check (auth.uid() = user_id);
  
  -- Note: quest generation (AI-created quests) should run through a
  -- server-side service role (bypassing RLS), not the client directly,
  -- since the AI Admin — not the player — creates quest rows.

  # Nexora System — Phase 1 Build Prompt
  ### For use in Claude Code. Reference alongside NEXORA_SPEC.md and phase1_schema.sql.
  
  ---
  
  ## 0. MCPs to use during this build
  
  - **Figma MCP / Stitch** — generate/refine screen UI from the visual spec below (glassmorphic panels, gold/charcoal palette). Use for producing component-level designs before wiring logic, especially for the Status Window and Data Input popup, which have the most visual complexity.
  - **Supabase MCP** — query/inspect the live schema directly (`users`, `user_stats`, `quests`) instead of guessing column names; use it to verify RLS policies are behaving correctly as you wire up auth and writes.
  - Standard Claude Code file/edit tools for the actual React Native/Expo implementation.
  
  Workflow suggestion: use Figma/Stitch MCP to produce the visual layer for a screen → Claude Code implements it as a React Native component → Supabase MCP to verify the data layer connects correctly.
  
  ---
  
  ## 1. Scope for this phase
  
  Build the 6-screen onboarding flow + a static (non-AI-generated-quest) Status Window. No quest generation, no Shop, no Auction — those are later phases. Auth + data writes to `users` and `user_stats` per `phase1_schema.sql`.
  
  ---
  
  ## 2. Screen-by-screen spec
  
  ### Screen 1 — Initialize System
  - Full-screen, near-black background (`#0A0A0B` or similar deep charcoal, not pure black)
  - Center: Nexora System logo/emblem icon (gold, glowing outline) — use a simple geometric sigil/rune mark, not a literal wordmark, for the boot moment
  - Below logo: thin gold horizontal line that "draws itself" left-to-right (simple width animation, 0 → 100%, ~800ms)
  - Typewriter text beneath: `INITIALIZING SYSTEM...`
  - **Transition to Screen 2:** fade to black (200ms), then fade in Screen 2 (300ms) — no slide
  
  ### Screen 2 — Username & Password
  - Glassmorphic panel (see §4) centered on screen, ~80% width
  - Two input fields inside the panel:
    - Username field — icon: simple person/silhouette glyph (gold outline, 20px), left-aligned inside field
    - Password field — icon: padlock glyph (gold outline, 20px), left-aligned inside field, with a show/hide eye icon on the right
  - Field style: transparent background, thin gold-bottom-border only (not full box border) — reads more "terminal input" than "form field"
  - Primary action button below fields: `ENTER THE SYSTEM` — gold-filled button, sharp corners (2px radius max), subtle glow on press
  - **Transition to Screen 3:** on successful submit, panel fades out (200ms) while background begins the loading sequence
  
  ### Screen 3 — Loading / Registration Sequence
  - Same near-black background, no panel — just centered typewriter text, one line at a time, each line fully typed before the next begins:
    1. `Registering user...` (pause ~600ms)
    2. `Logging in user...` (pause ~600ms)
    3. `User logged in.` (pause ~800ms)
  - All three lines fade out together (300ms)
  - Fade in: `Welcome to the Nexora System, [username]` — larger text, gold, centered (hold ~1200ms)
  - Fade out welcome text (300ms), fade in Status Window teaser:
    - Same Status Window layout as Screen 6, but every value field shows `??` (Name, Title, Level, Trait, Stats) instead of real data
    - This state persists ~2 seconds before Screen 4 appears
  
  ### Screen 4 — Data Input Popup
  - Appears as a modal over the (still-visible, blurred) `??` Status Window — background blur increases when this modal is present (ties into the modal-blur rule from the style guide)
  - Glassmorphic panel, scrollable if needed, fields:
    - Real name (text input)
    - Main Quest / yearly goal (text input, larger field — this is the important one)
    - Body weight (numeric input)
    - Time available per day (numeric input, minutes, or a simple slider)
    - Equipment/resource constraints (optional text input)
  - Icon per field: goal field gets a small target/flag glyph, body weight gets a small scale glyph, time gets a small hourglass glyph — keep consistent gold-outline icon language throughout
  - Submit button: `SYNC TO SYSTEM`
  - **Transition to Screen 5:** modal fades out, background Status Window (still showing `??`) remains visible underneath
  
  ### Screen 5 — Syncing
  - Typewriter text: `Syncing user to the System...`
  - Subtle animated element while this resolves — e.g. the gold border of the (still-visible, blurred) Status Window panel pulses/breathes, suggesting it's being written to
  - Hold ~1500ms (real duration should match actual backend write time, don't fake a shorter wait than the real request takes)
  
  ### Screen 6 — Status Window (Populated)
  - Same exact layout/position as the Screen 3 teaser — this is the payoff, so the transition should feel like the `??` values are "resolving" into real ones, not a hard screen swap
  - Each `??` individually flips/fades into its real value (Name, Title = "Novice", Level = "1", Stats = starting values) — stagger these slightly (50-100ms apart) rather than all at once, for a "coming online" feel
  - This is now the app's home screen going forward
  
  ---
  
  ## 3. Icon set needed for onboarding (v1 minimum)
  
  All icons: gold outline style, consistent stroke weight (~1.5-2px), simple/geometric, no fill except on active/selected states.
  
  - System emblem/sigil (Screen 1)
  - Username glyph (person silhouette)
  - Password glyph (padlock)
  - Show/hide password (eye / eye-slash)
  - Goal/target glyph (Screen 4)
  - Body weight/scale glyph (Screen 4)
  - Time/hourglass glyph (Screen 4)
  - Rank letter badge frame (for the top-right Rank display on Status Window)
  - Quest tier markers — three small distinct glyphs for Main/Side/Daily line items on the Status Window (e.g. a crown/banner for Main, a waypoint for Side, a simple checkmark-circle for Daily)
  
  ---
  
  ## 4. Glassmorphic panel spec
  
  Since the style guide calls for **sharp geometric corners**, this is glassmorphism adapted to that language — not the soft-rounded version common in most UI kits.
  
  - Background: semi-transparent charcoal, ~15-20% opacity over blur (`rgba(20, 20, 24, 0.65)` as a starting point, tune visually)
  - Backdrop blur: ~20px
  - Border: 1px solid gold at low opacity (`rgba(212, 175, 55, 0.4)` — adjust to actual gold hex once picked), NOT a thick border
  - Corner radius: small, 2-4px max — enough to not look like a harsh box, not enough to read as "soft app card"
  - Optional: very subtle inner glow/shadow using the gold accent at low opacity, to reinforce the "System projection" feel without literal holography
  - Panels should sit ABOVE a blurred/dimmed version of whatever's behind them (background blur increases whenever a panel/modal is present, consistent across the whole app going forward, not just onboarding)
  
  ---
  
  ## 5. Data writes for this phase
  
  On Screen 4 submit → Screen 6 populated, the following should be written to Supabase (per `phase1_schema.sql`):
  
  - `users` row: `username`, `name`, `body_weight`, `time_available_minutes`, `level = 1`, `exp = 0`, `rank = 'E'`
  - `user_stats` rows: initial baseline stat values (starting values TBD — can default to a flat low value like 5 across all stats for Phase 1, real stat-derivation logic from goal category comes later)
  - Main Quest itself: for Phase 1, it's fine to just store the raw goal text on the `users` row or as a single `quests` row with `tier = 'main'`, `status = 'active'` — no AI quest generation logic yet, just persist what the user typed
  
  ---
  
  ## 6. Explicitly not in this phase
  
  - No AI-generated quests (Daily/Side quest generation is Phase 2)
  - No Shop/Auction
  - No real verification logic — this phase is purely onboarding + static Status Window

  Context
  Nexora System is a solo-built "Life OS" RPG app (React Native/Expo + Supabase). The founder has an empty repo and two specs: NEXORA_SPEC.md (full product vision, already saved) and a detailed Phase 1 build prompt (onboarding flow + static Status Window). Phase 1 scope: 6-screen onboarding sequence, auth, and a populated Status Window home screen, writing to Supabase per a schema that doesn't exist yet. What is available: a stitch-design skill, but its companion conversion skill (react-components) targets Vite/web React, not React Native/Expo — so Stitch can't cleanly round-trip into RN components here.
