  # Implement Governance Token
  
  Per the locked design (Governance_Token_Design.pdf, all three
  decisions confirmed by Dmitry): build the ERC-20 token, fixed-window
  freeze mechanism, and staking-based fee distribution. Token name still
  TBD — use a placeholder (e.g. `NexaiGovToken`) that's easy to rename
  later.
  
  ## Part 1: Token Contract
  
  - Standard ERC-20, **100M fixed supply**, minted once at deployment
    (or via the vesting/allocation flow in Part 4) — no ongoing mint
    function after initial distribution
  - **5% max wallet cap**: override the transfer hook (`_update` in
    OZ v5, or `_beforeTokenTransfer` depending on the OZ version already
    in use elsewhere in this repo — check and match) to revert any
    transfer that would push a recipient's balance above 5% of total
    supply. Exempt the distribution/vesting contracts themselves from
    this cap during initial setup (they'll temporarily hold more than
    5% before distributing) — use a clear allowlist, not a hack.
  
  ## Part 2: Fixed-Window Voting Freeze
  
  - Separate, single-responsibility mechanism from Part 3 (staking) —
    do not let these share state or logic
  - Flow: a vote opens for N days (parameterize N, don't hardcode) →
    any token used to vote gets frozen (cannot transfer) for exactly
    that window → unfreezes automatically when the window closes
  - This is specifically to prevent flash-loan-style vote manipulation
    (borrow tokens right before a vote, vote, return immediately after)
    — voting power should be based on frozen balance during the window,
    not balance at an easily-manipulated instant
  - Voting power itself: linear (1 token = 1 vote), per the locked
    decision — no sqrt/quadratic math anywhere in this contract
  
  ## Part 3: Staking-Based Fee Distribution
  
  - Deliberately separate from Part 2 — staking does NOT grant voting
    power, and voting/freezing does NOT affect staking rewards
  - Reference Reserve Protocol's `StakingVault`/`UnstakingManager`
    pattern (found earlier in the trial) for the reward-per-token-staked
    accumulator approach — reuse/adapt rather than inventing new
    distribution math from scratch, consistent with this project's
    general approach
  - Fee revenue (0.2% mint, 0.2% redeem, 1.5% annual management from
    $NEXAI, per the original concept doc) flows in; staked holders earn
    proportional to their staked balance; unstaked holders earn nothing
  
  ## Part 4: Vesting / Allocation
  
  - Use OpenZeppelin's `VestingWallet` (or equivalent audited primitive)
    per beneficiary — do not write custom vesting logic
  - Allocations per the concept doc: 40% Public/IDO (no lock), 15% Team
    (2yr vesting, 6mo cliff), 20% Treasury/DAO, 15% Ecosystem (2yr
    vesting), 10% Liquidity (6mo lock)
  - Team and Ecosystem allocations need actual `VestingWallet` instances
    deployed per beneficiary; Liquidity's "6mo lock" and Public/IDO's
    "no lock" are simpler (a lock contract or direct transfer
    respectively) — don't force everything through the vesting pattern
    if it doesn't fit
  
  ## Explicitly Out of Scope (Flag, Don't Build)
  
  - IDO sale mechanism itself (contract, whitelist, payment flow) —
    still undesigned, flag if this comes up rather than improvising
  - Quorum denominator definition — flag as a decision needed during
    implementation, per Dmitry's note that some items will be decided
    as we build
  - AI Grants Fund allocation process/destination
  - Referral tracking (cross-cutting into NexaiCore/NexaiZapper)
  - World ID integration (deferred to a future upgrade)
  - Treasury/DAO multisig wiring — note whether it should reuse the
    existing NexaiCore multisig/timelock, but don't build new admin
    infrastructure here without confirming first

  Standard rigor, same as the rest of this project:
  - Fixed supply correctness, 5% wallet cap enforcement (including the
    exemption allowlist behaving correctly)
  - Freeze mechanism: tokens used to vote are frozen for exactly the
    window, unfreeze correctly after, cannot be double-used to vote
    again while frozen
  - Staking: reward accumulation math correct under sequential
    stake/unstake/claim operations from multiple users (same rigor as
    the `NexaiCore` multi-depositor tests)
  - Confirm staking and freeze/voting are genuinely independent —
    staking doesn't affect voting power, freezing doesn't affect
    staking rewards
  - Vesting: correct cliff/linear-release behavior per beneficiary

  // SPDX-License-Identifier: MIT
  pragma solidity ^0.8.20;
  
  import {ERC20} from "@openzeppelin/contracts/token/ERC20/ERC20.sol";
  import {AccessControl} from "@openzeppelin/contracts/access/AccessControl.sol";
  
  /**
   * @title NexaiGovToken
   * @dev Placeholder name for NEXAI's governance token - final name/symbol still TBD per
   * Governance_Token_Design.pdf; only this contract name and the ERC20 name/symbol below
   * need to change to rename later, nothing else references them by string.
   *
   * Fixed 100M supply, minted once in the constructor - no mint function exists
   * afterwards. Enforces a 5% max-wallet cap on regular holders via `_beforeTokenTransfer`
   * (this repo is on OpenZeppelin 4.9.x, which hooks transfers here rather than through
   * v5's `_update`). Custody contracts (vesting wallets, locks, the staking vault, the
   * voting-freeze escrow) are exempted via an explicit allowlist - they legitimately hold
   * more than 5% of supply while servicing many end users, which is not the concentration
   * risk the cap is meant to catch.
   */
  contract NexaiGovToken is ERC20, AccessControl {
      bytes32 public constant ADMIN_ROLE = keccak256("ADMIN_ROLE");
  
      uint256 public constant TOTAL_SUPPLY = 100_000_000 * 10 ** 18;
      uint256 public constant MAX_WALLET_BPS = 500; // 5% = 500 / 10_000
      uint256 public constant BPS_DENOMINATOR = 10_000;
  
      mapping(address => bool) public isCapExempt;
  
      event CapExemptionSet(address indexed account, bool exempt);
  
      constructor() ERC20("NEXAI Governance Token", "NEXGOV") {
          _grantRole(DEFAULT_ADMIN_ROLE, msg.sender);
          _grantRole(ADMIN_ROLE, msg.sender);
  
          // Exempt the deployer up front so the full fixed supply can be minted to it and
          // distributed from there without immediately tripping the cap on itself.
          isCapExempt[msg.sender] = true;
          emit CapExemptionSet(msg.sender, true);
  
          _mint(msg.sender, TOTAL_SUPPLY);
      }
  
      /// @dev Toggle the 5% cap exemption for `account`. Meant for custody contracts
      /// (vesting/locks/staking/voting-escrow) that legitimately aggregate large balances
      /// on behalf of many end users - not for regular holders.
      function setCapExempt(address account, bool exempt) external onlyRole(ADMIN_ROLE) {
          isCapExempt[account] = exempt;
          emit CapExemptionSet(account, exempt);
      }
  
      function maxWallet() public view returns (uint256) {
          return (TOTAL_SUPPLY * MAX_WALLET_BPS) / BPS_DENOMINATOR;
      }
  
      function _beforeTokenTransfer(address from, address to, uint256 amount) internal override {
          super._beforeTokenTransfer(from, to, amount);
  
          // Burns (to == address(0)) are never capped. Mints/transfers into an exempt
          // address (e.g. a custody contract) skip the check entirely.
          if (to != address(0) && !isCapExempt[to]) {
              require(balanceOf(to) + amount <= maxWallet(), "NexaiGovToken: exceeds max wallet cap");
          }
      }
  }

  /**
   * @title DeploySafes
   * @dev Deploys two Safe v1.4.1 multisig wallets sharing the same 3 owners but
   * different thresholds, per NEXAI_Multisig_Timelock_Design.pdf:
   *  - AdminSafe (2-of-3): proposer on the TimelockController (see DeployTimelock.s.sol)
   *  - GuardianSafe (1-of-3): instant pause on NexaiCore + canceller on the Timelock

   * sub-threshold, so two deployments with the same owner set is the clean way to get
   * both a 2-of-3 governance path and a 1-of-3 emergency path using only audited,
   * unmodified Safe contracts.
   *
   * Addresses below are Safe's canonical v1.4.1 deployment on Base Sepolia, verified
   * against the safe-global/safe-deployments registry (not assumed).

      /// given signer keys. Returns false rather than reverting the whole test if the
      /// Safe rejects the transaction (e.g. the inner call reverted) - several tests
      /// deliberately exercise that failure path.
      function _execSafeTx(address safe, uint256[] memory signerKeys, address to, bytes memory data)
          internal
          returns (bool success)
      {
          bytes memory signatures = _buildSignatures(safe, signerKeys, to, data);
          try ISafe(safe).execTransaction(to, 0, data, 0, 0, 0, 0, address(0), payable(address(0)), signatures) returns (
              bool ok
          ) {
              success = ok;
          } catch {
              success = false;
          }
      }
  
      /// @dev Builds a Safe-format signature blob: signs the transaction's EIP-712 hash
      /// with each given key, ordered by increasing signer address (a hard Safe
      /// requirement for its signature check).
      function _buildSignatures(address safe, uint256[] memory signerKeys, address to, bytes memory data)
          internal
          view
          returns (bytes memory signatures)
      {
          uint256 txNonce = ISafe(safe).nonce();
          bytes32 txHash =
              ISafe(safe).getTransactionHash(to, 0, data, 0, 0, 0, 0, address(0), address(0), txNonce);
  
          address[] memory signers = new address[](signerKeys.length);
          for (uint256 i = 0; i < signerKeys.length; i++) {
              signers[i] = vm.addr(signerKeys[i]);
          }
          for (uint256 i = 1; i < signerKeys.length; i++) {
              uint256 keyToInsert = signerKeys[i];
              address addrToInsert = signers[i];
              uint256 j = i;
              while (j > 0 && signers[j - 1] > addrToInsert) {
                  signers[j] = signers[j - 1];
                  signerKeys[j] = signerKeys[j - 1];
                  j--;
              }
              signers[j] = addrToInsert;
              signerKeys[j] = keyToInsert;
          }
  
          for (uint256 i = 0; i < signerKeys.length; i++) {
              (uint8 v, bytes32 r, bytes32 s) = vm.sign(signerKeys[i], txHash);
              signatures = abi.encodePacked(signatures, r, s, v);
          }
      }

  # Implement Vesting via Sablier (Part 4, Governance Token)
  
  Per Dmitry's decision: use Sablier (not Hedgey, not custom-built
  vesting) for Team and Ecosystem allocation vesting — specifically for
  cancelable streams, which matter given the forfeiture-on-early-
  departure clause in the co-founder agreement.
  
  ## Step 0: Verify the actual deployed address first
  
  Do not assume or reuse a hardcoded address from memory/training data.
  Sablier's protocol has moved to a unified `SablierLockup` contract
  (replacing the older separate LockupLinear/LockupDynamic/LockupTranched
  contracts) — confirm current architecture and the actual deployed
  address on **Base Sepolia** directly from Sablier's own sources before
  writing anything:
  - Official docs deployments page: https://docs.sablier.com (Lockup
    deployments section)
  - Or their `sablier-labs/sdk` package/repo, which programmatically
    tracks addresses per chain/release
  
  Comment the source and exact page/method used to find the address, so
  it's independently checkable later — same standard as every other
  external address used in this project (Safe, Uniswap, Chainlink).
  
  ## Requirements
  
  - Integrate via Sablier's standard interface (`ISablierLockup` or
    equivalent per whatever the verified current version's interface is
    named) — do not vendor Sablier's contracts into this repo; call the
    already-deployed, already-audited singleton directly, same pattern
    used for Safe and Uniswap
  - **Team allocation** (15% of supply): create a cancelable linear
    vesting stream per team member, 2-year duration, 6-month cliff
  - **Ecosystem & Partnerships allocation** (15% of supply): cancelable
    linear vesting stream per recipient, 2-year duration, no cliff
    specified in the concept doc — confirm this assumption is correct
    before building, flag if ambiguous rather than guessing
  - **Liquidity allocation** (10%): 6-month lock — check whether this
    is better handled via a simple Sablier stream with a 6-month
    cliff/no-release-before-then, or a separate simpler lock mechanism;
    don't force it through vesting logic if a plain timelock is more
    appropriate
  - **Public/IDO allocation** (40%): no lock — this doesn't need Sablier
    at all, just a direct transfer once the (still undesigned) IDO
    mechanism exists; don't build anything for this portion now beyond
    reserving/earmarking the tokens
  - Cancelable streams: confirm how cancellation works in the verified
    current Sablier version (who can cancel, what happens to
    already-vested vs. not-yet-vested tokens on cancellation) and make
    sure that behavior actually matches the forfeiture clause's intent
    — flag to the client if there's a mismatch rather than assuming it

  Sablier's Lockup contract will hold tokens on behalf of many
  beneficiaries at once — likely exceeding 5% of supply in aggregate.
  This needs the same cap-exemption pattern already built into
  `NexaiGovToken.sol` (`setCapExempt`). Exempt Sablier's verified Lockup
  contract address before funding any streams, same as the exemption
  already used for other custody contracts (staking, voting freeze).

  - Confirm streams are created correctly with the right duration/cliff
    per allocation type
  - Confirm cancellation behavior matches what's actually documented for
    the verified Sablier version — don't assume, test against the real

  - Confirm the cap-exemption is correctly applied before any stream
    funding is attempted
  
  ## Notes
  
  - Testnet only, no mainnet
  - If the verified Sablier address/interface differs meaningfully from
    what's assumed here, stop and report back rather than guessing —
    same standard applied throughout this project
